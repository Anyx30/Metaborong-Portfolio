Content first or Design first?

The order shapes how the whole build goes. Here are the three realistic approaches for the Metaborong revamp.

① SEO Content Architecture
→ 12 service pages mapped to keywords
→ URL structure, H1/H2/schema plan
→ Blog pillar + cluster map
② Design wraps the content
→ Layouts designed around real copy
→ CTAs, headings, hero — all real

A — Content First

Write SEO-optimized copy for every page first. Design is then built around real content — no lorem ipsum, no guessing at heading lengths or CTA placement.

Pros

  • SEO structure baked in from day one
  • Design fits real copy — no layout surprises at handoff
  • Recommended by the strategy doc for Phase 1

Cons

  • Slower start — no visuals until content is drafted
  • Content decisions can block design work
① Design System + Pages
→ Import your Claude design tokens
→ Build homepage + service page shells
→ Placeholder content throughout
② Fill with SEO content
→ Drop real copy into built templates
→ Adjust layouts where content doesn't fit

B — Design First

Import your design system and build the visual framework first. Content gets filled in after. Faster to see something real on screen, but risks layout rework when real SEO copy doesn't fit the placeholder shapes.

Pros

  • Visual momentum — you can see the site quickly
  • Design system established early

Cons

  • Layouts often break when real content replaces lorem ipsum
  • SEO structure can conflict with visual decisions already made
Simultaneous tracks:
Track 1 — Content
→ SEO copy for homepage + 3 core service pages
Track 2 — Design system
→ Tokens, typography, color, components
→ Merge: build page by page with real content

C — Parallel (Recommended)

Run content and design as two short parallel tracks. First: nail the SEO content architecture (page map, URL structure, keyword→page mapping). Second: import your design system and establish tokens + base components. Then merge — build each page with real content from the start.

Pros

  • Neither track blocks the other
  • Design system ready when content is ready
  • Pages built once, correctly — no rework

Cons

  • Requires deciding content architecture upfront
  • Slightly more planning before first visible output
