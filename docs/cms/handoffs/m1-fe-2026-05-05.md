## 7. Known gaps / open questions

1. **`proxy.ts` modification is a scope-stretch.** The dispatch's "DO NOT"
   list says "anything outside /admin and /blog routes — homepage work is
   unrelated." `proxy.ts` is project-root infrastructure, not a route. I
   modified it because:
   - Next.js 16 App Router exposes no pathname API to server components.
     The dispatch explicitly requires `redirect('/admin/login?next={pathname}')`
     from the layout for *any* nested admin route. Without a header
     injection, the layout can only reliably redirect from `/admin` itself.
   - The change is purely additive: prepends an `x-pathname` request
     header onto the existing `NextResponse.next({...})` call. The geo /
     consent cookie logic that already lives in `proxy.ts` is untouched.
   - I considered adding a separate `middleware.ts` but Next.js 16
     forbids both files coexisting. Folding into `proxy.ts` was the only
     option short of breaking the dispatch's gate-every-nested-admin-route
     requirement.

   **Flagging for TPM review.** If the policy is strict, the gate can be
   downgraded to "redirect from `/admin` only" and the proxy change
   reverted — but that loses the deep-link `next=` behavior.

2. **`lib/auth.ts` is BE-owned and currently lives on this branch via the
   orchestration's BE-auth commit (`01292e5`), not via my work.** My
   layout consumes the public surface (`getAdminSession()` returning
   `AdminSession | null`); per the dispatch, if BE wasn't merged I was to
   write a stub. In practice the orchestration surfaced BE's real
   implementation onto this branch, so no stub was committed.

3. **Typecheck on this branch fails at BE-owned imports.** Running
   `pnpm exec tsc --noEmit` reports four errors, all in `app/api/admin/login/route.ts`
   and `lib/api.ts` (BE files in commit `01292e5`), all of the form
   "Cannot find module './db/client'" / "./blog-schema". Those files
   (`db/client.ts`, `db/schema.ts`, `lib/blog-schema.ts`) are part of the
   BE deliverable and currently sit untracked in the working tree but are
   not on this branch's commits. **No FE-owned file emits a typecheck
   error.** When BE's `cms/m1-be-plumbing` branch lands first (per
   workflow §1 merge order), `cms-dev` will have those files and the
   FE merge will compile clean.

4. **Test scope expansion (staged but uncommitted in
   `docs/cms/agent-prompts.md`) was NOT implemented.** I observed a

5. **Browser-tested?** No — I have no browser environment in this

6. **Login form's CSRF cookie expectation.** The `lib/api-client.ts`
   throws `CsrfMissingError` when `mb_csrf` cookie is absent on a
   non-GET request. This is fine for every admin request *except* the
   very first `POST /api/admin/login` — that's the call that establishes
   the session and issues the cookie. The login form catches
   `CsrfMissingError` and shows "Could not establish a session — please
   refresh and try again." This presumes Backend issues `mb_csrf` either
   on the GET that renders `/admin/login` (preferred — BE check) or via
   a separate seeding mechanism. **Open question for BE:** confirm the
   first-load CSRF cookie strategy.

7. **30-day session messaging.** The login page subhead says "Your
   session lasts 30 days" — sourced from PRD §5.1. If BE's session TTL

## 8. Manual verification (3-5 step golden path)

Assumes BE branch (`cms/m1-be-plumbing`) is merged into the local working
copy first (per workflow §5 step 2), `.env.local` is configured per BE's
example file, and Postgres / migrations are running.

1. **Auth gate.** `pnpm dev` → visit `http://localhost:3000/admin` in a
   private window. Expected: 302 to `/admin/login?next=%2Fadmin`. Visit
   `http://localhost:3000/admin/posts/some-id` directly. Expected: 302
   to `/admin/login?next=%2Fadmin%2Fposts%2Fsome-id`.
2. **Login happy path.** On `/admin/login`, type the seeded admin email
   + password from `.env.local`. Press Enter. Expected: successful POST
   round-trip; `mb_admin_session` (HttpOnly) and `mb_csrf` cookies set;
   browser lands on `/admin` (or the `next=` path you arrived from).
3. **Login bad-creds.** Sign out, try a wrong password. Expected:
   inline error region (red text, role="alert") shows the API's `error`
   field; email input retains its value, password input is cleared and
   focused. No console error, no raw stack trace.
4. **Empty submit.** Sign out, leave both fields empty, click Sign in.
   Expected: no network request fires; inline error reads "Email is
   required."; focus moves to email input.
5. **Sign out + back-button.** Sign in, then click Sign out (top-right).
   Expected: redirect to `/admin/login`. Press the browser back button.
   Expected: the request to `/admin` 302s back to `/admin/login` (the
   layout-level gate re-evaluates and the cookie is gone).

If any step shows raw "undefined" or a stack trace in the UI, that is a
defect — file against FE.
