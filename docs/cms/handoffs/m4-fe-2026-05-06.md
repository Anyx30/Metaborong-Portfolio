Server-only helpers added by this dispatch (consumed only by Server
Components, not Route Handlers):

- `getImageById(id)` — used by `app/admin/posts/[id]/page.tsx` to
  resolve the cover + og image rows for the form's initial state.

  post in one round-trip; the resolver passed to `<PostView />` reads
  from the resulting Map.

`api.postMultipart<T>` (new in `lib/api-client.ts`) is the single FE
boundary for multipart uploads with progress; XHR-backed because
`fetch()` does not surface `progress` events for upload bodies.

   Open `/blog/<slug>` in an incognito window. The cover renders at

## 8. Known gaps / open questions

1. **Focal-point visual editor not in this dispatch.** Picker tiles and
   the BlockRenderer / PostView both honor the row's `focal_x` /
   `focal_y` as `objectPosition`, and the BE's `PATCH
   /api/admin/images/[id]` accepts `{ focal_x, focal_y }`. What's
   missing is the draggable-dot UI — so today every image starts (and
   stays) at the (0.5, 0.5) default. The dispatch listed focal point as
   an FE edge case but I treated it as a small follow-up rather than
   block the milestone; happy to add a dot-and-overlay control to the
   Library tile and/or the upload preview if Tester wants it inside M4.

2. **`api.patch` with empty body used as "GET by id" in
   `ImageNodeView`.** Per BE handoff §4: PATCH with empty body is a
   no-op that returns the existing row. We use that as a single-row
   read because the BE intentionally did not ship `GET
   /api/admin/images/[id]` (out of scope for M4). Not a defect; just a
   subtle reuse worth flagging in case BE ever changes the empty-body
   semantics.

3. **Paste-URL fetch is CORS-bound.** The clipboard-URL path uses
   `fetch(url, { mode: 'cors' })` to grab bytes before uploading. Most
   image hosts (Imgur, Twitter image CDN) block cross-origin fetches.
   Surfaces a clear "Could not fetch the pasted URL" message; admins
   can drag-drop or save-then-upload as a workaround. Treating this as
   a 90 % path improvement, not v1 critical.

4. **No syntax highlighting on code blocks.** Carryover from M3; M4
   doesn't change that.

5. **E2E spec not executed in this dispatch.** Same caveat as the M3
   handoff. The spec compiles; Tester should run it under their
   seeded admin DB + provisioned `BLOB_READ_WRITE_TOKEN`.

6. **`next/image` remote pattern allowlist.** Already configured by BE
   (`next.config.ts` `images.remotePatterns`) for
   `*.public.blob.vercel-storage.com`. If a future Vercel Blob change
   alters the host pattern, every image on the site would 404 through
   `next/image` and we'd need to update the allowlist; flagged here so
   the next maintainer doesn't have to dig for it.

7. **React Compiler hint warnings.** `npx eslint` surfaces a handful of
   "set-state-in-effect" / "preserve-manual-memoization" warnings on
   the new picker and node-view modules. Same pattern is already
   present in `edit-post-form.tsx`, `editor.tsx`, and other M3 files;
   treated as stylistic React-Compiler hints rather than bugs (build
   + tests + typecheck all pass). Worth a sweep alongside the existing
   warnings in a future hygiene PR.
