## 3. Item 1 — SaveState flicker

### 3.1 Root cause

Two surfaces displayed save status at once:

1. The role=status `<SaveIndicator />` in the action bar
   (`components/admin/edit-post-form.tsx:713`).
2. A separate green "Saved Xs ago" toast banner directly under the
   action bar, gated on `savedToastAt > 0 && save.kind === 'saved'`.

The toast had **no auto-dismiss**, so it pinned forever after the
first save and visibly disagreed with the live indicator on every
subsequent autosave. SaveIndicator itself never returned to `idle`
either — `setSave({kind:'saved'})` left the state pinned to 'saved'
with a per-second timestamp interval ticking up "Xs ago" indefinitely.

The user's report — *"the tiptap editor keeps loading after we write
something on this, which is bit irritating"* — read as flicker,
because each autosave re-triggered the indicator's spinner-then-saved
transition while the stale toast underneath remained "Saved 47s ago".

The dispatch's hypothesis ("autosave debounce vs save-state
transition aren't aligned") was second-order; the live debounce
itself was already correct (one autosave fire per 2s of quiet).

### 3.2 Fix summary

- Removed the duplicate toast banner outright. SaveIndicator is the
  single source of truth.
- Added a `useEffect` that schedules `setSave({kind:'idle'})` 2s
  after entering 'saved'. Cleanup clears the timer when the state
  changes for any other reason (e.g. user starts a new save), so a
  fresh save's `setSave({kind:'saving'})` cancels the lingering-fade
  cleanly.
- Removed `SaveIndicator`'s `force`-render interval — with 'saved'
  capped at ≤2s, "just now" → "1s ago" is the only visible tick, no
  point holding a setInterval open.
- 'saving' entry is unchanged: still gated by the 2s autosave
  debounce, so rapid keystrokes never surface an intermediate
  "Saving…".

### 3.3 Before / after observable behavior

**Before:**

- Type something. Wait 2s. Indicator spinner shows "Saving…" briefly,
  then "Saved just now". Toast banner appears below: "Saved just now".
- Keep typing. Indicator stays at "Saved Xs ago", ticking up. Toast
  banner ALSO stays at "Saved Xs ago", same number ticking up.
- 2s after next quiet → indicator briefly flips to "Saving…" again,
  toast still says the old "Saved Xs ago" until the indicator lands
  on the new "Saved just now". User reads the disagreement as
  flicker.

**After:**

- Type. Indicator shows "Unsaved changes" (idle + dirty).
- 2s quiet → "Saving…" with spinner.
- Save lands → "Saved just now".
- 2s later → indicator fades to "Up to date" (or back to "Unsaved
  changes" if the user has been typing during the linger window).
- No second banner. No stale "Saved 47s ago" pinning underneath.

### 3.4 Tests added

`components/admin/edit-post-form.test.tsx` — two new cases (12/12
pass on this branch):

1. **`SaveState lifecycle: idle → saving → saved (lingers ≥2s) →
   idle`** — uses `vi.useFakeTimers`. Stages the PATCH to resolve on
   a controlled promise so the 'saving' state is observable; advances
   timers 1s into the linger window, asserts 'saved' still showing;
   advances another 1.5s, asserts 'saved' gone and "Up to date"
   present.
2. **`only one "Saving…" indicator appears per real save (no flicker
   on rapid keystrokes)`** — fires five rapid title edits inside the
   2s autosave window; asserts the 'Saving…' indicator never
   surfaces during typing; advances past the autosave threshold and
   asserts exactly one PATCH was issued.

The "Saving…" string lives both in the role=status indicator AND on
the disabled Save button label — `getAllByText` rather than
`getByText` to keep the assertions duplicate-tolerant.

## 4. Item 2 — Tiptap split/paste fresh-id plugin

### 4.1 Root cause

ProseMirror's built-in `splitBlock` command (Enter inside a heading
or paragraph) calls `tr.split(pos)` which copies the parent node's
attrs verbatim onto the new sibling — `id` included. Likewise
copy/paste within the editor reproduces the source id on the pasted
node. Both produce duplicate ids that downstream consumers had to
work around:

- `lib/editor/serialize.ts editorStateToBlocks` (commit `4196e03`)
  freshens duplicates at serialize time so saved Block[] are
  guaranteed unique.
- `components/blog/post-view.tsx dedupeBlockKeys` (commit `d23d1df`)
  freshens render keys at read time so pre-fix posts already in the
  DB don't trip React's "two children with the same key" guard.

Both stay in place — the right v1.5 fix is to stop creating
duplicates in the editor in the first place, not to remove the
defensive shims that protect data already on disk.

### 4.2 Fix summary

New `components/admin/editor/unique-ids.ts` exports a Tiptap
`Extension` whose sole job is to register a ProseMirror `Plugin`
with an `appendTransaction` hook. After every doc-changed
transaction the hook walks `newState.doc`'s top-level children:

- Tracks ids already claimed by an earlier sibling.
- For any node whose id is missing/null OR already claimed,
  generates a fresh uuid via `crypto.randomUUID()` (with the same
  Math.random fallback the serializer uses).
- Returns one transaction per fix, all stamped
  `tr.setMeta('addToHistory', false)` so undo/redo doesn't replay
  the rename as a separate user action.

`addProseMirrorPlugins` is the only Tiptap extension surface used —
no schema changes, no command changes. The walk is O(n) over
top-level children (block-level mb-* nodes are direct children of
doc; `MbListItem`, the only nested mb-* node, carries no `id` attr
by design).

The extension is wired into `components/admin/editor/editor.tsx`
alongside `MB_NODES` in the Tiptap `extensions` array. No call-site
change beyond that.

### 4.3 Before / after observable behavior

**Before (post-`4196e03` serializer):**

- User has a heading "Why we shipped" with `id=abc-123`. Caret
  mid-word, presses Enter. Tiptap splits → two headings, both
  `id=abc-123` in the editor's state.
- Block[] save: serializer freshens the second to `id=def-456` on
  the way out. Saved row is clean, but the editor's state still
  carries the duplicate until the next round-trip
  (post → server → setBlocks(res.post.content_json)).
- Live preview pane reads from the editor's state in-flight,
  including the brief duplicate. PostView's `dedupeBlockKeys`
  silently re-keys the offending node.
- Manual repro: open devtools, watch the editor's `editor.getJSON()`
  — duplicates visibly present until next autosave round-trip.

**After:**

- Same sequence — Enter mid-heading splits into two headings. The
  appendTransaction hook fires immediately on the same dispatch and
  rewrites the right-hand half's id to a fresh uuid. By the time
  React's render commit runs, the editor state already has unique
  ids.
- Round-trip is unchanged — serializer still emits unique ids; just
  rarely has anything to fix. Defensive shim in PostView still
  catches pre-fix posts.

### 4.4 Tests added

`components/admin/editor/unique-ids.test.tsx` (`@vitest-environment
happy-dom`) — 4/4 pass:

1. **Paste-style duplication** — Two paragraphs sharing
   `bbbbbbbb-…`. After a nudge transaction (insert+delete one char to
   trigger appendTransaction), the first occurrence keeps its id,
   the second is a fresh uuid.
2. **`splitBlock` (Enter inside a heading)** — Single heading
   `BeforeAfter` with `cccccccc-…`. `editor.commands.setTextSelection
   (7); editor.commands.splitBlock()` → two headings, both with
   non-null ids, distinct from each other; first half retains the
   seed id, right half is fresh.
3. **No-op when ids already unique** — Two paragraphs with distinct
   ids `dddddddd-…` and `eeeeeeee-…`. Nudge transaction. Doc still
   contains both ids verbatim, in order.
4. **Missing id filled in** — Single paragraph with no `id` attr in
   the seed JSON (parser defaults `id` to `null` per the schema).
   Nudge transaction. The orphan now carries a fresh uuid.

The harness sets `trailingNode: false` on StarterKit because its
default-type fallback inserts a trailing `mbHeading` on any edit
(when `paragraph` is disabled in the schema, the schema's first
registered node — `MbHeading` — becomes the default content type).
That auto-insertion is unrelated to the plugin under test and was
just polluting assertions.

## 5. Item 3 — M1 e2e page.goBack flake

### 5.1 Root cause (suspected, not confirmed)

`e2e/m1-auth.spec.ts:53` — the "browser back lands on login"
assertion fails because `page.goBack()` resolves to `about:blank`
under Playwright + Next.js App Router. The same flake reproduces
across M1, M2, M3, and M4 Tester reports; all four flagged it
out-of-scope and recommended the same `page.goto('/admin')`
work-around.

Suspected cause: the Sign Out flow ends with a server-side redirect
(middleware) to `/admin/login`, while the prior `/admin` page was
reached via App Router's client-side navigation
(`router.push`/`replace`). Mixing those two history-update mechanisms
leaves Playwright's history-back waiter unable to observe the
resulting URL — the page resolves but Playwright's navigation
promise sees `about:blank` and times out.

### 5.2 Fix attempted

Root-cause work requires a runnable e2e: Neon-compatible POSTGRES_URL
(the local docker `postgres:16` does NOT work, per M1 report HIGH 1)
PLUS `E2E_ADMIN_EMAIL` + `E2E_ADMIN_PASSWORD` exported into the
shell PLUS the matching `ADMIN_PASSWORD_HASH` seeded in `.env.local`.
This branch's environment had none of those, so any try-it-and-see
fix would land unverified.

Per the dispatch's 30-minute budget rule, marked the test
**`test.fixme()`** (deliberately not `test.skip()`) so it stays
visible in reports as known-broken with a tracking commitment, and
left a multi-line `TODO(v1.6)` enumerating three concrete approaches
the next Tester should try in priority order:

1. `await page.evaluate(() => window.history.back())` followed by
   `page.waitForURL(/\/admin\/login/, { timeout: 5000 })`. Lets us
   bypass Playwright's goBack waiter and instead wait on the URL
   we actually want.
2. `await page.context().clearCookies()` before the back step. Makes
   the gate's redirect path unambiguous (no session = guaranteed
   bounce regardless of timing).
3. Replace `goBack` with `page.goto('/admin')`. Meaningfully narrows
   intent (back-button vs direct nav) and overlaps the existing
   `unauthenticated visit to /admin redirects` test in the same
   describe block, so it shrinks coverage — only fall back to this
   if (1) and (2) both fail.

### 5.3 Before / after observable behavior

**Before:** Test runs (when creds are set) and fails with
`expect(page).toHaveURL(/\/admin\/login/)` because page URL is
`about:blank`. Subsequent specs in the same serial mode are
unaffected (M2/M3 specs use their own describe blocks).

**After:** Test reported as `fixme` in the Playwright reporter (the
`-` prefix in `--reporter=list` output, "skipped (1)" in totals).
The other three M1 specs continue to pass.

### 5.4 Tests added

None. This item is intentionally a non-fix that stops papering over
a known infrastructure flake.
