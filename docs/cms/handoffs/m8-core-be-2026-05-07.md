## 7. Known gaps / open questions

1. **Magic-byte sniff happens AFTER the bounded multipart read, not before.** The dispatch's spec wording was *"sniffs the first N bytes BEFORE buffering the whole body"*. The current pipeline reads the bounded body (capped at `IMAGE_MAX_BYTES + 64KB ≈ 8.06 MB`), parses multipart to extract the file part, **then** sniffs. The OOM concern the spec was guarding against is mitigated by the bounded read — the worker can never buffer more than ~8 MB of attacker bytes regardless of whether the file part is genuine. To literally sniff "first N bytes" before any read would require a streaming multipart parser, which is a much larger change (today we lean on `Request.formData()` for parsing). Flagged as a v1.5 follow-up if true streaming is needed; not blocking for v1.

2. **Login rate-limit prune.** `login_attempts` rows older than the 15-minute window aren't actively pruned — the M1 BE handoff noted this and deferred to "M8 periodic prune". This BE pass kept scope tight to the spoofing fix; the prune job is still outstanding. Low risk today (table size grows linearly with failed-login traffic, which is bounded by the 5-attempts-per-IP-per-window cap) but worth a small `DELETE FROM login_attempts WHERE attempted_at < now() - interval '15 minutes'` call somewhere — either a Vercel Cron or piggy-back on a successful-login path. Flag for v1.5+.

3. **`@vercel/postgres` deprecation.** Carried forward from M1 / M4. Untouched in this dispatch.

4. **Branch-context race during this dispatch.** Documented in §1 above; the working state is clean but worth a Tester eye on `git log cms/m8-core-fe-polish --oneline` to confirm only FE polish commits live there.

5. **No coverage for x-real-ip on Vercel.** The new `clientIp()` returns x-real-ip on Vercel only as a tertiary fallback when XFF is empty after parsing. That branch is exercised in tests but I haven't verified what real Vercel infra puts in those headers — the implementation reads the spec, not a real production capture. Worth a one-off `console.log(clientIp(req))` on a Vercel preview the next time someone deploys, just to confirm the buckets behave as expected.
