## 5. Endpoints implemented

Reference §2.3 of `docs/cms/agent-prompts.md`.

| Method | Path | Behavior |
|---|---|---|
| `POST` | `/api/admin/posts/[id]/ai-readiness` | requireAdmin + requireCsrf → 503 if MCP disabled → load post → 409 `POST_NOT_PUBLISHED` if not published → content-hash cache hit returns `cached:true` → 429 if 30 attempts already in the last hour for this admin → `scanUrl()` → persist (score, band, report jsonb, content_hash, checked_at) → 200 `{ score, band, report, cached:false, scannedAt }`. |
| `GET` | `/api/admin/posts/[id]/ai-readiness` | requireAdmin → returns the last stored `{ score, band, report, cached:true, scannedAt }`, or 404 `NOT_SCORED` if `ai_readiness_score` is null. |

Response shape on success (both methods):

```ts
{
  score: number,                  // overallScore rounded to int 0..100
  band: 'strong' | 'adequate' | 'weak',
  report: AiReadinessReport,      // VerseOdin shape — see §6
  cached: boolean,
  scannedAt: string               // ISO 8601
}
```

Error envelope is the §2.2 `{ error, code, field? }` shape. New code
constants (added to `errorCodeSchema`):

- `POST_NOT_PUBLISHED` — 409, draft-scoring deferred to v1.6.
- `NOT_SCORED` — 404, GET on a never-scanned post.

## 6. Wire shape — `AiReadinessReport`

Mirrors the VerseOdin tool response (probed live 2026-05-08 — see §Appendix
for the verbatim curl response):

```ts
{
  overallScore: number,           // 0..100
  pageScore: number,              // 0..100
  domainScore: number,            // 0..100
  domainReputationBonus: number,
  metadata: {
    title: string | null,
    description: string | null,
    analyzedAt: string,           // ISO 8601 from VerseOdin
  },
  checks: Array<{
    id: 'robots-txt' | 'sitemap' | 'llms-txt' | 'heading-structure'
       | 'readability' | 'meta-tags' | 'semantic-html' | 'accessibility',
    label: string,
    status: 'pass' | 'warning' | 'fail',
    score: number,                // 0..100
    scope: 'page' | 'domain',
    details: string,
    recommendation: string,
  }>,
}
```

Schema is `aiReadinessReportSchema` in `lib/blog-schema.ts`. The top-level
object uses Zod `.passthrough()` so unknown fields (future VerseOdin
additions) round-trip without breaking validation. The `id` field on
each check is a strict enum of the 8 known ids — adding a new id
upstream will require a schema bump (intentional: typed FE chip rows
should move in lock-step with the contract).

## 7. Edge cases handled

Mapped against §4 "EDGE CASES YOU OWN":

- ✅ 401 vs 403 vs 422: 401 on missing/invalid session; 403 on missing/wrong CSRF; 409 `POST_NOT_PUBLISHED`; 404 `NOT_FOUND` on missing post; 404 `NOT_SCORED` on never-scanned GET.
- ✅ AI readiness MCP disabled: any of the 3 env vars unset → 503 `MCP_DISABLED`. Never leak which one. Treat whitespace-only as missing.
- ✅ AI readiness caching: `sha256(canonicalUrl + '|' + post.updated_at)`; cache hit returns the stored row verbatim with `cached:true` if `ai_readiness_checked_at` is < 1h old. Skips the rate-limit increment so a same-content re-check is free.
- ✅ AI readiness rate limit: 30 / hour per admin via `ai_readiness_attempts`; 429 with `Retry-After: 3600` on overflow. Increment happens **after** the cache check and **before** the upstream call.
- ✅ AI readiness timeout: 10s `AbortController` cap; `AbortError` → `McpTimeoutError` → 504 `MCP_UPSTREAM_ERROR`. No score persisted on timeout.
- ✅ Auth fail upstream: 401/403 from VerseOdin → `McpAuthError` → 503 `MCP_DISABLED` (so the FE drawer's "service misconfigured" UX surfaces and the operator gets a server-side log line with the upstream status).
- ✅ Malformed payload: JSON parse fail / Zod fail → `McpInvalidPayloadError` → 502 `MCP_UPSTREAM_ERROR`.
- ✅ All POST require CSRF before any DB read.
- ✅ Cookies / sessions: re-uses existing `requireAdmin` and `requireCsrf` from `lib/auth.ts`. No changes to cookie config.
- ✅ Token never leaked: error responses use generic messages; only `console.error` logs include the upstream status (not the bearer).

## 8. How to test locally without the Frontend

Bring the dev server up against a local Postgres on port 5433 (the
`mb-test-pg` container that the BE/FE agents use) **and** the live
VerseOdin endpoint. Then login to mint cookies:

```bash
# 1) Login (mints mb_admin_session + mb_csrf)
curl -sS -c /tmp/mb-cookies -X POST http://localhost:3000/api/admin/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@metaborong.test","password":"<dev-password>"}'

# Extract CSRF for follow-up calls
CSRF=$(awk '/mb_csrf/{print $7}' /tmp/mb-cookies)

# 2) Create a draft, publish it (M2 endpoints)
POST_ID=$(curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -H 'Content-Type: application/json' \
  -X POST http://localhost:3000/api/admin/posts \
  -d '{"title":"AI readiness probe"}' | jq -r '.post.id')
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -X POST http://localhost:3000/api/admin/posts/$POST_ID/publish

# 3) Happy path — POST ai-readiness (fresh scan)
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -X POST http://localhost:3000/api/admin/posts/$POST_ID/ai-readiness | jq

# Expected: { score, band: 'strong'|'adequate'|'weak', report: {…}, cached:false, scannedAt }

# 4) Cache hit — same post, no edits
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -X POST http://localhost:3000/api/admin/posts/$POST_ID/ai-readiness | jq '.cached'
# Expected: true

# 5) Error: draft-scoring (unpublish then POST)
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -X POST http://localhost:3000/api/admin/posts/$POST_ID/unpublish
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -i -X POST http://localhost:3000/api/admin/posts/$POST_ID/ai-readiness
# Expected: HTTP/1.1 409, body { error, code: 'POST_NOT_PUBLISHED', … }

# 6) Error: missing CSRF → 403
curl -sS -b /tmp/mb-cookies \
  -i -X POST http://localhost:3000/api/admin/posts/$POST_ID/ai-readiness
# Expected: HTTP/1.1 403, code: 'CSRF_FAILED'
```

For the MCP-disabled path, unset any of the 3 env vars, restart `next dev`,
and call POST: `503` with `code: 'MCP_DISABLED'`.

## 9. Known gaps / open questions

### 9.1 FE-shape contract change (drawer + dashboard pill)

`lib/blog-schema.ts` previously stubbed the report as
`{ suggestions: AiReadinessSuggestion[] }`. **That stub is gone.** The
real shape now ships per §6 above. FE callers that rendered
`report.suggestions` need to switch to `report.checks` and adapt:

- `report.checks[i]` carries `{ id, label, status, score, scope, details, recommendation }`. Use `recommendation` as the "what to do next" body copy.
- Per-check `status` is `'pass' | 'warning' | 'fail'` (lowercase).
- Per-check `scope` is `'page' | 'domain'` (lowercase). The pre-existing FE WIP at `lib/ai-readiness/ui-types.ts` typed scope as `'PAGE' | 'DOMAIN'` (uppercase) — that needs to flip.
- VerseOdin checks are **not block-scoped** in v1, so the legacy
  "click suggestion → scroll to block" affordance from PRD §5.10 step 4
  is deferred. The FE drawer should render the recommendation text but
  not link to a block id.
- `band` is now strictly `'strong' | 'adequate' | 'weak'` (lowercase).
  The dashboard pill / button pill should `.toUpperCase()` for display
  (existing `bandPalette()` already lowercases for matching).

### 9.3 Pre-publish scoring (v1.6)

PRD §10 leaves draft scoring open: VerseOdin needs a public URL, so v1.5
restricts scoring to published posts. v1.6 will add a tokenized preview
URL (`/blog/preview/<token>/`) that VerseOdin can fetch with a one-shot
`X-Preview-Token` header. Out of scope here.

### 9.4 BE-internal vs wire-exposed schema split

Per the dispatch prompt, `verseOdinReportSchema` (BE-internal name) and
`aiReadinessReportSchema` (wire-exposed name) are kept aligned as a
single schema for now, exported once from `lib/blog-schema.ts` and
re-exported from `lib/ai-readiness/types.ts`. If the upstream shape
diverges from what we want to persist (e.g. we want to drop `metadata`
from the wire response but keep it server-side), we'll split them then.

### 9.5 Drizzle snapshot for 0002

Skipped (see §4). Regenerate cleanly with `pnpm db:generate` against a
fresh DB if/when M8 needs another migration.

### 9.6 Race on the `attempts` insert

The 30/hour gate uses count-then-insert without a transaction, so two
concurrent calls from the same admin could both pass the count check and
then both insert. The PRD §5.10 quota is advisory — burning one extra
call past the cap won't break VerseOdin or the cache — so a transaction
felt like overkill here. Mirrors the same trade-off in
`lib/api/login/route.ts`. If the BE later moves to high-concurrency
admin shells, wrap both queries in a single statement
(`INSERT … WHERE (SELECT count(*) …) < 30 RETURNING …`).
