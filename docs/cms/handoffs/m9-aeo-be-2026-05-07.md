## 3. New library API

```ts
// lib/seo.ts
export function faqPageSchema(post: Post): Record<string, unknown> | null
export function howToSchema(post: Post): Record<string, unknown> | null
export function speakableSchema(post: Post): Record<string, unknown> | null
export function deriveStepName(block: Block): string

// lib/blocks-to-md.ts
export function blockToPlainText(block: Block): string
```

Return-type rationale: I matched `articleSchema`'s `Record<string, unknown>` over the prompt's literal `object | null` because the M5-core call sites already index into the JSON-LD object (`out.publisher['@type']`, etc.), and `Record<string, unknown>` keeps that ergonomic. Either would compile.

### Behavior summary

| Builder | Triggers off | Returns null when | Output `@type` |
|---------|--------------|-------------------|----------------|
| `faqPageSchema`   | `block.type === 'faq'`                | zero faq blocks   | `FAQPage` with `mainEntity: Question[]` |
| `howToSchema`     | `block.role === 'step'`               | < 3 step blocks   | `HowTo` with `step: HowToStep[]`, positions 1..N |
| `speakableSchema` | `block.role === 'tldr' \|\| 'intro'`   | zero such blocks  | `WebPage` with `speakable: SpeakableSpecification` |

`howToSchema` omits the `description` key entirely when both `post.excerpt` and `post.meta_description` are null/empty — no `description: undefined` ever serializes. The threshold of 3 mirrors Google Rich Results' guidance that single-step "how-to" markup is treated as low-quality.

`speakableSchema` builds CSS selectors of the form `[data-block-id="<id>"]`. **The public block renderer must emit a `data-block-id` attribute on each rendered block for these selectors to resolve.** If FE has not yet wired this attribute, Speakable will validate but match nothing — see §7.1.

## 4. Edge cases handled

Mapped to "EDGE CASES YOU OWN" in the dispatch prompt:

| Case | Handling |
|------|----------|
| Empty / null fields never emit `undefined` | `howToSchema.description` is conditionally assigned (`if (description) out.description = …`) so the key is absent when both fall-throughs are null. No other field defaults to `undefined`. |
| HTML escaping in user content | All user strings round-trip through `JSON.stringify` at the call site (the FE / `<script type="application/ld+json">` site stringifies the returned object). I verified `blockToPlainText` does not introduce raw HTML — it returns plain Unicode strings, no markup. |
| Long content not truncated for FAQ answer / HowTo step text | `acceptedAnswer.text` is the raw `b.data.answer`; `HowToStep.text` is `blockToPlainText(b)` which is the full block body. Truncation only happens for `HowToStep.name` via `deriveStepName` (80-char cap), per spec. |
| HowTo `position` is 1-indexed and preserves content_json order | `steps.map((b, i) => ({ position: i + 1, … }))`. Tested with a 4-block content_json (3 steps + 1 non-step interleaved); positions come out 1, 2, 3 in content order. |
| FAQ with empty answer still emits | The block-type filter is `b.type === 'faq'` regardless of `answer`. The Zod `faqBlockSchema` rejects `answer: ''` at the wire-shape layer, so this only fires for objects constructed in tests / future renderer paths that bypass parse — but the contract said "don't gate on data quality", so `faqPageSchema` itself doesn't filter. |
| `deriveStepName` first-sentence extraction | Splits on `.`, `?`, `!` via `/^([^.!?]+)[.!?]/`. Tested with all three terminators. Trailing punctuation is stripped from the captured group via the same pattern (the punctuation char is matched outside the group). |
| `deriveStepName` 80-char truncation at word boundary | `truncateAtWordBoundary` mirrors the heuristic in `lib/blocks-to-md.ts` (`deriveTextDescription`) — break on the last space within the cut window if it's past the 60% mark, else hard-cut. No trailing ellipsis (this is a "name", not a description). Tested with a 249-char single-sentence paragraph: result is ≤ 80 chars and ends on a word boundary. |
| `deriveStepName` non-text fallback | Image → `alt` text. FAQ → `question`. Code & empty paragraph → literal `'Step'`. List → first item. Heading / paragraph / quote / callout / key-takeaway → first sentence of `text`. Tested per case. |

## 7. Known gaps / open questions

1. **Renderer must emit `data-block-id`.** `speakableSchema` produces selectors of the form `[data-block-id="<block.id>"]`. The public block renderer (FE-owned, `components/blog/post-view.tsx` and friends) needs to emit this attribute on each rendered block for the selectors to actually anchor the speakable region. If the FE M9-AEO dispatch hasn't already added this, it should — flag against FE if missing.
2. **`@id` namespacing.** The current Speakable output is a top-level `WebPage` JSON-LD without an explicit `@id`. Google's docs accept this, but if the page already emits an `Article` JSON-LD with `mainEntityOfPage: { @id: <url> }` (it does, from M5-core), some validators will complain about two `@type: 'WebPage'` shapes referring to the same canonical URL without disambiguation. If the Tester / FE flag this, the cleanest fix is to give Speakable an `@id: <canonical-url>#speakable` so the two graphs are explicitly distinct. Did not implement to avoid scope creep — confirm with TPM whether to add.
3. **HowTo threshold is hard-coded at 3.** Reasonable for v1.5, but if a future post has 2 well-formed steps + a clear sequence, the post still won't get HowTo markup. PRD doesn't define a threshold; spec from the dispatch said "fewer than 3 → null". Open to revisiting once we have telemetry on which posts qualify.
4. **`blockToPlainText` does not strip inline markdown.** Block `data.text` is stored as plain text in the editor's current shape (no inline markdown syntax inside paragraph blocks per `lib/blog-schema.ts`), so this is a no-op concern today. If a future editor change starts storing inline markdown (e.g. `**bold**`) inside `data.text`, the HowTo step `text` and `deriveStepName` outputs would carry those literal markers into JSON-LD. Worth a one-line note in the editor migration doc when that change lands.
5. **No FE call-site wiring done here, intentionally.** Per dispatch ("M9-AEO is library-only"). FE consumes the new builders in their parallel branch.
