## 4. Block ↔ Tiptap state mapping (§2.4 type → Tiptap node name)

The serializer at `lib/editor/serialize.ts` is the single transform between
the canonical `Block[]` (lib/blog-schema.ts) and the editor's ProseMirror
JSON tree. Round-trip identity is binding:

```
blocksToEditorState(editorStateToBlocks(state)) ≡ state
editorStateToBlocks(blocksToEditorState(blocks)) ≡ blocks
```

| Block.type      | Tiptap node name (mb-prefixed) | Atom? | Attrs                             | Inline content |
|-----------------|--------------------------------|-------|-----------------------------------|----------------|
| `heading`       | `mbHeading`                    | no    | `id, role, level, anchor`         | text + marks   |
| `paragraph`     | `mbParagraph`                  | no    | `id, role`                        | text + marks   |
| `image`         | `mbImage`                      | yes   | `id, role, imageId, alt, caption` | —              |
| `list`          | `mbList` (+ `mbListItem` children) | no | `id, role, ordered`               | items as children |
| `quote`         | `mbQuote`                      | no    | `id, role, cite`                  | text + marks   |
| `code`          | `mbCode`                       | no    | `id, role, lang`                  | text only      |
| `callout`       | `mbCallout`                    | no    | `id, role, tone`                  | text + marks   |
| `faq`           | `mbFaq`                        | yes   | `id, role, question, answer`      | —              |
| `key-takeaway`  | `mbKeyTakeaway`                | no    | `id, role`                        | text + marks   |

Tiptap StarterKit is loaded with all built-in block-level nodes disabled
(`heading`, `paragraph`, `bulletList`, `orderedList`, `listItem`,
`blockquote`, `codeBlock`, `horizontalRule` all `false`) so the editor's
schema only emits mb-prefixed nodes. `Document`, `Text`, `History`,
`Dropcursor`, `Gapcursor`, `HardBreak`, plus the inline marks
(bold/italic/strike/code) stay on for keyboard editing.

## 11. Defect fix — heading-level dropdown not visible to Playwright

**Reported:** TPM, immediately after the prior session stopped — the
M3 e2e spec (`e2e/m3-block-editor.spec.ts`) failed at
`page.getByLabel(/heading level/i).first().selectOption('3')` with
"element is not visible" while the locator resolved to the right
`<select>`. Vitest stayed 247/247 green.

### 11.1 Root cause

In `components/admin/editor/node-views.tsx` the `HeadingNodeView`
wrapped the level `<select>` in:

```tsx
<div
  contentEditable={false}
  className="absolute -top-[18px] left-0 hidden items-center gap-1 group-hover:flex group-focus-within:flex"
>
```

The intent was a discreet badge that surfaces only on hover or when
the heading has focus. That logic is broken under Tiptap's DOM
topology:

- The NodeViewWrapper carries `class="group"`. Its `:focus-within`
  pseudo-class only matches when **a descendant** is focused.
- Tiptap's contentEditable host is the editor's outer `<div
  class="mb-tiptap-doc">` — an **ancestor** of every NodeViewWrapper.
  When the user types into the heading, `document.activeElement` is
  that outer host, not the heading wrapper. So the heading's
  `:focus-within` is always `false`.
- Playwright never hovers (`selectOption` clicks the element directly,
  doesn't sweep over its parent), so `:hover` never fires either.
- Result: `display: none` on the wrapper, `<select>` is "not visible",
  Playwright waits 30s and times out.

The Vitest tests rendered `HeadingNodeView` directly via
`render(<HeadingNodeView {...} />)`, where the wrapper is the test's
top-level container — so `:focus-within` worked the moment the
select got focus, hiding the bug from the unit harness.

### 11.2 Fix summary

Single one-line change in `node-views.tsx`: drop `hidden ...
group-hover:flex group-focus-within:flex` and pin the wrapper to
`flex` so the H2..H6 dropdown is always rendered. Also nudged the
top offset from `-18px` to `-16px` to sit a little less aggressively
above the heading text. Added a comment explaining the topology
constraint so the next agent doesn't re-introduce the hover trick.

This is the right product call too — the heading-level control is
the heading block's primary affordance and should be discoverable
without a hover. UX cost is minor (small mono-uppercase pill above
each heading); SEO + a11y benefit is real.

### 12.2 MEDIUM 2 — list-style toggle has polluted accessible names

**Root cause.** `components/admin/editor/inspector.tsx`'s `Field`
helper renders `<label><span>{label}</span>{children}</label>` —
correct for form-control children (`<input>`, `<select>`,
`<textarea>`) but wrong for arbitrary `<button>`s. The List block's
two toggle buttons were the only `Field` consumers shaped that way.
HTML's `<label>`-implicit-association rule meant each button's
accessible name was computed as `label.textContent`, so screen readers
heard the Unordered button as **"List style Ordered"** with
`aria-pressed="true": Unordered` — the field label + the *other*
button's text leaked into both buttons' a11y names.

**Resolution.** `ListFields` no longer reuses `<Field>`. It now
renders its own `<div>` with a styled `<span id={useId()}>` for the
label text, and groups the two buttons in
`<div role="group" aria-labelledby={labelId}>`. `<Field>` itself is
unchanged — every other consumer of it (Role, Heading Level,
Anchor, Tone, Image fields, Citation, Language, FAQ Q/A) is a real
form control, so the existing implicit-label semantics still fit.
Net diff: ~12 lines in one component file, no change to visual
output.

**Verification — accessibility tree (Playwright probe).** Probe at
`/tmp/m3-list-a11y.mjs` against the running dev server, on a draft
seeded via API with one list block:
```

Ordered button count:   1
Unordered aria-pressed: true              ← default state (ordered=false)
Ordered aria-pressed:   false
group "List style" count: 1               ← role=group with accessible name
group inner-button names: [ 'Unordered', 'Ordered' ]
after click → Unordered aria-pressed: false
after click → Ordered   aria-pressed: true
```
`getByRole('button', { name: 'Unordered', exact: true })` resolves
exactly one node — the Tester report's smoking gun (zero matches
before the fix) is gone.

**No new component test added.** Inspector has no `*.test.tsx`

### 12.4 HIGH 3 — `e2e/m3-block-editor.spec.ts` flakes on fresh `.next` + truncated DB

**Surfaced by.** TPM verification — Tester's session reported 3 / 3
green for the M3 spec, but TPM could not reproduce on a clean state
(kill dev server, `rm -rf .next`, TRUNCATE posts). On that fresh
state, test 1 timed out at 30s on
`getByPlaceholder(/what is geo/i).fill(...)` (the FAQ question
input, line 121 of the original spec).

**Root cause — three reinforcing layers.** The failure screenshot
showed the editor canvas with `"/paragraph"`, `"/faq"`, etc. typed
as plain text inside heading blocks instead of the slash menu
inserting them. Tracing through:

1. `selectOption('3')` on the heading-level dropdown leaves focus
   on the `<select>`. Subsequent keystrokes don't reach
   `editor.view.dom`, where `slash-menu.tsx`'s `keydown` listener for
   `/` is bound. Without that listener firing, the menu doesn't
   open and `/` is just typed into whatever block currently holds
   the caret. The `End` + `Enter` between heading and paragraph
   inserts compounded the problem: with a custom `MbHeading` (no
   `splitBlockAfterEnter` keymap, unlike StarterKit's `Heading`),
   ProseMirror's default `splitBlock` on Enter creates *another
   heading*, not a paragraph, so the test ended up with a chain of
   empty H2/H3 blocks each holding the typed slash command as
   plain text.
2. The `mb_consent` cookie is null on Playwright's fresh contexts,
   so `<ConsentBanner>` mounts at `z-[60]` in the lower-right of
   the page. It doesn't visually overlap the editor canvas, but it
   adds `Accept` / `Reject` buttons to the document and sat
   stubbornly on top of every failure screenshot — TPM was right
   that "the consent banner makes [the focus drift] obvious".
3. `getByText('We chipped away at the deadline.')` resolves in
   *two* places: the editor canvas and the live preview pane
   (after its 150ms debounce). Strict-mode `.click()` either errors
   on multi-match or — worse, silently — clicks the preview's
   non-editable copy. Either way the caret is not restored to the
   contentEditable surface, so the very next slash never opens
   the menu.

**Resolution — test-only, no production change.** Three small
tweaks, all in `e2e/m3-block-editor.spec.ts`:

1. **`signIn` pre-sets `mb_consent` via `addInitScript`**:
   ```ts
   await page.context().addInitScript(() => {
     document.cookie = 'mb_consent=rejected; path=/; SameSite=Lax'
   })
   await page.goto('/admin/login')
   ```
   `addInitScript` runs before any page JS, so
   `useConsent`'s `useState(() => readConsent())` reads
   `'rejected'` on first render and `<ConsentBanner>` short-circuits
   to `null`. The `<ConsentRevokePill>` renders instead, which
   doesn't intercept anything.
2. **Restore caret after `selectOption('3')`** by clicking the
   heading text *scoped to the editor canvas*:
   ```ts
   const canvas = page.getByTestId('editor-canvas')
   await canvas.getByText('Why we shipped').click()
   await page.keyboard.press('End')
   ```
   The scoping prevents Playwright from clicking the preview pane's
   matching H3 and dropping the caret outside the contentEditable.
3. **Drop the `keyboard.press('End') + keyboard.press('Enter')`
   between blocks.** The slash menu's `deleteRange` +
   `insertContent` path inserts the new block as a sibling and
   lands the caret inside it — a manual Enter is unnecessary and
   actively harmful here, because Tiptap's default `splitBlock` on
   `MbHeading` produces another heading rather than the paragraph
   the test expected. The same canvas-scoped focus restoration is
   applied before the FAQ insert (post-paragraph) and before the
   final paragraph click that wires `role='tldr'` via the
   inspector.

No production code changed — `MbHeading` keeping default
`splitBlock` semantics is a real UX issue (Notion / Linear demote
to paragraph on Enter) but the dispatch was explicit that test-only
fixes are preferred when they suffice. Filed as a future-work
item in §8 below could be added if the team wants Notion-style
behavior, but that is editor-design work, not a regression.
