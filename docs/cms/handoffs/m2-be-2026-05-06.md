## 4. Edge cases handled

Mapped to "EDGE CASES YOU OWN (M2 binding)" from the dispatch:

| Edge case                                                                                | Status |
|------------------------------------------------------------------------------------------|--------|
| Slug uniqueness race-safe (UNIQUE constraint + duplicate-key catch → SLUG_CONFLICT)      | PASS — both POST /posts and PATCH /posts/[id] catch `isUniqueViolation()` and translate to 422 SLUG_CONFLICT with field='slug'. Verified by `route.test.ts > returns 422 SLUG_CONFLICT on duplicate slug` in both files. |
| Slug immutability after first publish (PATCH 422 with VALIDATION_FAILED + field='slug')  | PASS — `PATCH /posts/[id]` checks `existing.published_at !== null` before allowing slug change. Verified: `rejects slug change once post is published` and `allows slug change for a never-published post`. Error message: `'Slug is immutable after first publish.'` |
| Publish flow (transaction: status='published' + published_at if null + revalidate paths) | PASS — `COALESCE(published_at, $now)` in a single UPDATE means the publish is atomic without an explicit BEGIN/COMMIT. `revalidatePath('/blog')` and `revalidatePath('/blog/' + slug, 'page')` fire BEFORE returning. Republish preserves the original timestamp. |
| Unpublish (status='draft' + published_at PRESERVED + revalidate)                         | PASS — verified by `flips status to draft and PRESERVES published_at`. The route deliberately omits `published_at` from the SET clause. |
| Delete (hard delete + revalidate + DO NOT cascade images)                                | PASS — schema FKs on `posts.cover_image_id` / `posts.og_image_id` use `ON DELETE SET NULL` (M1); the DELETE handler issues no DELETE on `images`. Revalidates only when the deleted post was `published`. |
| 404 vs 403 (admin-only [id] route returns 404 on missing OR non-uuid)                    | PASS — UUID-shape check before the DB read keeps probing from leaking schema details. Admin sees drafts; public helpers `getPostBySlug` and `listPublishedPosts` filter `status='published'` so drafts are NEVER exposed to the public. |
| All non-GET routes verify CSRF before any DB read                                        | PASS — every non-GET handler calls `requireCsrf(req)` first thing, then `requireAdmin(req)`. Verified by `403 without CSRF header` tests on POST /posts, PATCH /posts/[id], DELETE /posts/[id], POST /posts/[id]/publish, POST /posts/[id]/unpublish. |
| Body validation via Zod, .strict() on PATCH                                              | PASS — `patchPostBodySchema` is `.strict()`, so `{ status: 'published' }` (or any unknown field) returns 422 VALIDATION_FAILED. `createPostBodySchema` is also `.strict()`. |
| Errors normalize to §2.2 envelope                                                        | PASS — every non-2xx flows through `errorResponse()` from `lib/api.ts`. |
| Geo-variant resolver emits same regions as `lib/use-geo` contract (US/EU/OTHER)          | PASS — `resolveRegion()` returns the contract's `GeoRegion` type. EU bucket is EU-27 + UK + EFTA. `mergeVariant()` returns the base post unchanged for OTHER and never exposes other regions' variants. |

### PATCH happy + slug-immutable + slug-conflict

```sh
# Happy: update title and tags on a draft
curl -i -b /tmp/mb.txt -X PATCH \
  -H 'Content-Type: application/json' \
  -H "X-CSRF-Token: $CSRF" \
  -d '{"title":"Hello, DeFi!","tags":["web3","defi"]}' \
  http://localhost:3000/api/admin/posts/$POST_ID
# expect: 200, post.title updated

# Slug-conflict: try to take a slug another post owns
curl -i -b /tmp/mb.txt -X PATCH \
  -H 'Content-Type: application/json' \
  -H "X-CSRF-Token: $CSRF" \
  -d '{"slug":"some-other-posts-slug"}' \
  http://localhost:3000/api/admin/posts/$POST_ID
# expect: 422, { "error": "...", "code": "SLUG_CONFLICT", "field": "slug" }

# Slug-immutable: publish the post first, then try to rename
curl -i -b /tmp/mb.txt -X POST -H "X-CSRF-Token: $CSRF" \
  http://localhost:3000/api/admin/posts/$POST_ID/publish
curl -i -b /tmp/mb.txt -X PATCH \
  -H 'Content-Type: application/json' \
  -H "X-CSRF-Token: $CSRF" \
  -d '{"slug":"renamed"}' \
  http://localhost:3000/api/admin/posts/$POST_ID
# expect: 422, { "error": "Slug is immutable after first publish.", "code": "VALIDATION_FAILED", "field": "slug" }
```

## 7. Known gaps / open questions

- **Geo-variant resolver: editor doesn't validate the `OTHER` key.** `geoVariantsSchema` already restricts variant keys to `US` / `EU`, so a PATCH that sends `{ geo_variants: { OTHER: {...} } }` 422s correctly. The renderer collapses `OTHER` to base. No follow-up needed for M2; M6 will likely add per-region UI hints in the editor.
- **Tag filter form choice:** `listPublishedPosts` uses `$tag = ANY(tags)` instead of `tags @> ARRAY[$tag]` — pg-mem (used in tests) serializes single-element JS arrays differently from real PG and chokes on the `@>` form. Both forms use the GIN index in real PG. Not user-visible.
- **`listPublishedPosts` total via row scan:** v1 uses a bounded id-only scan to compute `total` and `hasMore` — fine for small catalogs. PRD §6.6 already calls out keyset pagination as the path past ~1000 posts.
- **Republish revalidation timing:** the publish handler revalidates after the SQL UPDATE returns. Next.js revalidation is best-effort; if it ever silently fails the next public request still re-renders within `export const revalidate = 60`. M5 ships the renderer; observability for revalidation failures is M8.
- **`canonical_url` on PATCH:** validated as a strict URL — passing `canonical_url: ''` returns 422. Editors who want to clear it should send `canonical_url: null`. Tester should confirm FE wires this correctly; the BE contract is already correct.
- **Slug auto-generation collisions:** `POST /api/admin/posts` doesn't auto-suffix on slug collisions today — if `slugifyTitle` produces a slug another post already owns, the response is 422 SLUG_CONFLICT and the FE has to retry with an explicit slug. PRD doesn't mandate auto-suffix; flagging as a v1.1 nice-to-have.
- **`db/test-utils.ts` diverged enough from M1 that it's worth a fresh pair of eyes from Tester.** Two pg-mem ↔ drizzle bridges now coexist (types/rowMode strip + object-rows-to-array conversion + synthesized `result.fields`). All M1 BE tests still pass; new M2 BE tests pass. But the patch is touching prototypes pg-mem owns, so a future pg-mem upgrade could reshape the surface.
- **Publish handler's published_at logic relies on COALESCE.** Real PG and pg-mem both honor it. If a future test driver doesn't, the alternative is a read-then-update pattern. Not a real-world concern for v1.
- **No M2 BE test for `lib/geo.ts mergeVariant` against a Post fetched from the DB end-to-end** — `lib/posts.test.ts > getPostBySlug merges US variant` covers this implicitly. Considered sufficient for M2; M6 will add wider coverage when the variant editor lands.
