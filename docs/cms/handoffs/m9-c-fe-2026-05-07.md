## 4. Edge cases handled

| Case                                                                                    | Handling |
|-----------------------------------------------------------------------------------------|----------|
| User edits US then switches to EU mid-session                                            | The form holds a single `variants` GeoVariants object that lives across tab switches; setVariantField reads/writes the active region without touching the other. |
| Empty override on a variant tab                                                          | `setVariantField` deletes the field key when value === ''; if the resulting region payload is empty, the region key itself is dropped before PATCH. |
| Other region carries an existing payload while user edits the active one               | The PATCH body is built from the local `variants` state, which always includes the unedited region untouched. Vitest case: "variant autosave preserves the existing other-region variant when editing one region". |
| Block overrides on an unsupported block type (list / code / faq)                         | Inspector panel renders an explicit "deferred to v1.6" hint instead of a write input. The form-level setter does not touch unsupported blocks. |
| User clears the only block override on a region that has no post-level fields           | `setBlockOverride` walks empty → block id removed → `block_overrides` removed → region payload empty → region removed from `geo_variants`. Vitest case: "block override write/clear". |
| Tiptap left in editable state when switching to a variant tab                            | `useEffect([editor, editable])` calls `editor.setEditable(editable)` on every change; preserves cursor + history (no remount). |
| Preview pane region selected manually                                                    | Manual selector change calls `setPreviewSynced(false)` so the lock breaks intentionally; user can re-sync via the toggle, which snaps `previewRegion` back to `activeVariant`. |
| Slug-locked AND on variant tab                                                           | Slug input renders disabled with the variant-tab hint priorities (variant copy wins; slug-lock copy still surfaced when on Base). |
| `localStorage` write throws in private mode                                              | Silently swallowed (matches the established split-pane pattern). The tab still works in-session; only persistence breaks. |
| Two posts open in two tabs                                                               | Persistence key includes the post id (`mb.editor.variant.<postId>`), so two tabs don't share state. |
| Title cleared on a variant tab                                                           | Empty = no override. The form's base-tab title validation (`Title is required.`) doesn't trigger — `state.title` is never empty on a variant tab because variant edits don't touch it. |

## 7. How to manually verify (5-step golden path)

Postgres on `:5433`, dev server on `:3000`, signed in as admin, at
least one draft post seeded.

1. **Tab strip + persistence.** Open the post edit screen. Confirm
   the Base · US · EU strip below the action bar with "Base" active.
   Click **US** — the tab highlights and `mb.editor.variant.<id>` in
   localStorage flips to `"US"`. Refresh the page — the US tab is
   re-selected.
2. **Override + chip + reset.** On US, type "US headline" into Title.
   The "OVERRIDES BASE" chip appears next to the field label. Click
   **Reset to base** — chip disappears, input clears. Type the title
   again, wait 2s for autosave; confirm the toast cycles
   `Saving… → Saved just now`. Switch to **Base** — the original base
   title is preserved unchanged.
3. **Block override.** Still on US, click into a paragraph in the
   editor. The right-rail inspector swaps to "Variant overrides · US"
   with a single-line "US text override" input. Type a few words; wait
   for autosave. The preview pane (when synced) shows the override
   applied to that block; switch the editor tab back to **Base** —
   block text reverts to base. Switch back to US — override returns.
4. **Other-region preservation.** While on US, click **EU** — the EU
   tab is empty (no overrides). Type a different EU headline; save.
   PATCH body in the network panel must contain BOTH `US` and `EU`
   keys with their respective payloads. Refresh; both stick.
5. **Dashboard chips.** Return to `/admin`. The post row now shows a
   small `US · EU` chip cluster. Edit the post and Reset both region
   chips; refresh `/admin` — chips disappear, `has_geo_variants`
   reverts to false.

## 8. How to verify the preview-pane sync

1. Open a post on the **US** tab. The preview pane header shows a
   blue "· Synced" pill and the Region selector reads "US".
2. Click the **· Synced** pill — it flips to "Sync with tab". Change
   Region to "EU" via the selector. The preview now resolves the EU
   variant overlay even though the editor stays on US.
3. Click "Sync with tab" — the preview snaps back to whatever the
   editor tab is.

## 9. Known gaps / open questions / out-of-scope

1. **`geo_variant_regions` on PostSummary is a (small) contract
   change.** This dispatch added `geo_variant_regions: ('US'|'EU')[]`
   to `lib/blog-schema.ts` and populated it in `lib/posts.ts:rowToSummary`.
   Backend agents should reconcile any other producer of `PostSummary`
   in M2 BE / M9-AEO BE that may have been emitting summaries inline
   (none found in this repo's grep, but flagging since `lib/blog-schema.ts`
   is the shared source). Empty payloads (e.g. `{ US: {} }`) intentionally
   don't set `has_geo_variants` true — matches the dispatch's
   "Empty variant string ≠ override" rule.
2. **`cta_label` / `cta_href` editor surface — deferred.** The schema
   reserves these on the variant payload but no consumer renders them
   in v1.5. The dispatch explicitly defers; flagged here so the v1.6
   plan can pick them up alongside a `<PostView>` CTA slot.
3. **Bulk variant editor / variant diff view — deferred (v1.6).** No
   surface in v1.5. The dashboard chip row is the closest thing to
   variant visibility the operator gets.
4. **Per-block override on lists / FAQs — deferred (v1.6).** The
   inspector renders an explicit "deferred" hint for these block
   types so the operator knows it's intentional, not a bug. The
   underlying `block_overrides` schema already accommodates `text`
   and `alt` only, so list-item-level overrides would also need a
   shape change.
5. **`mb.editor.variant.<id>` keys are not garbage-collected.** A
   power user editing many posts will accumulate small string entries
   forever. ~10 bytes/key — fine for v1, flag for v1.6 cleanup.
6. **No e2e spec authored in this dispatch.** The form is exercised
   by 5 new component tests (vitest + happy-dom). An e2e
   `e2e/m9-c.spec.ts` covering the variant golden path would be a
   sensible follow-up but exceeds this dispatch's scope. The v1
   admin DB seeding pattern from M5/M9-AEO is the right starting
   point.
7. **`pnpm lint` is broken at the project level** (predates this
   dispatch — `next lint` mis-parses the workspace path; same caveat
   carried by M9-GEO and v1-5-hygiene handoffs).
8. **The form-level test mocks `EditorShell` end-to-end.** This
   keeps Tiptap out of happy-dom, which is the established repo
   pattern (see comment in `edit-post-form.test.tsx`). The block
   override callback is exercised through hidden test buttons in
   the mock; the inspector panel itself is not yet covered by a
   component test. v1.6 follow-up: add a focused `inspector.test.tsx`
   that drives the variant override branch with a mocked editor.
9. **Slug / tags / author / images visually disable on variant tabs**
   but the underlying state still tracks any base edits the user
   made before switching. They flush on the next Base-tab autosave.
   Documented in the disabled hint copy ("switch to Base to edit").
