## 5. Backend endpoints consumed

Per `docs/cms/agent-prompts.md` §2.3 + the M7-BE handoff §3:

| Endpoint                                          | Method | When the FE calls it                                                  |
|---------------------------------------------------|--------|-----------------------------------------------------------------------|
| `/api/admin/posts/[id]/ai-readiness`              | GET    | Probe on `<AiReadinessButton>` mount when `post.ai_readiness_band` is null — 503 hides the button; any other response keeps it visible. |
| `/api/admin/posts/[id]/ai-readiness`              | POST   | Drawer auto-fires on open (no cached report), and the Re-scan button on the hero card. |
| `/api/admin/posts/[id]/publish`                   | POST   | `handlePublish` — unchanged from M2; M7-FE only reads the response to drive the soft-prompt. |

Wire shape consumed (canonical, per M7-BE):
```ts
{ score, band: 'strong'|'adequate'|'weak', report: AiReadinessReport, cached, scannedAt }
```
Errors consumed: `MCP_DISABLED` (503), `RATE_LIMITED` (429 + Retry-After),
`POST_NOT_PUBLISHED` (409), `NOT_SCORED` (404 — treated as "show button,
empty drawer state").

## 6. Edge cases handled (mapped to dispatch §EDGE CASES YOU OWN)

| Edge case                                                              | Implementation                                                                  |
|------------------------------------------------------------------------|---------------------------------------------------------------------------------|
| Button hidden when MCP disabled                                        | `useEffect` probe; `ApiError` with `status===503 && code==='MCP_DISABLED'` flips `probe.kind === 'hidden'`. |
| Button disabled with v1.5 tooltip on draft                             | `disabled={isDraft}` + `title` attr matches dispatch copy.                      |
| Score pill on the button when cached                                   | `<AiReadinessScorePill withSeparator />` rendered when `ai_readiness_score !== null`. |
| Drawer auto-fires POST when no cached report                           | `autoScanFiredRef` latch; resets on close so the soft-prompt re-fires it.       |
| Drawer 429 banner with Retry-After                                     | `parseRetryAfter` → `state.retryAt` → `HH:MM` formatted from a `Date`.          |
| Drawer 503 banner is admin-only (no token leak)                        | Banner copy: "Set the AI_READINESS_MCP_* env vars" — no upstream URL or token state. |
| Drawer 409 POST_NOT_PUBLISHED → "Publish now" CTA                      | `onPublishRequest` callback → parent's existing `handlePublish` (no duplicate validation). |
| Soft-prompt fires when `ai_readiness_checked_at` null OR > 24h old     | Inline `Date.now() - parse(checked_at) > 24 * 60 * 60 * 1000` after publish.    |

## 7. How to manually verify (golden path)

Pre-req: M7-BE working tree present (or merged on cms-dev) so the route
handler exists and `lib/blog-schema.ts` carries the canonical shapes.

1. `pnpm dev` and sign in at `/admin/login`.
2. Open any published post with no AI readiness score yet. The
   "Check AI readiness" button is visible in the top action bar
   (right of the status pill, left of Unpublish), without a score
   pill since none is cached.
3. Click "Check AI readiness". The drawer slides in from the right
   over the preview pane. A spinner appears for ~1–2s while the POST
   round-trips, then the hero card and 8 check cards render.
4. Click "Re-scan" on the hero. The hero score updates if VerseOdin
   returns a different number; otherwise the cached one (1h freshness)
   round-trips back instantly.
5. Close the drawer (× or Esc). The score pill is now visible on the
   button (`Check AI readiness · 87 · STRONG` for example).
6. Visit `/admin`. The post's row now shows a matching pill in the
   new score column.
7. Open a draft post. The button is disabled (greyed) with the
   tooltip "Score is only available after publish in v1.5".
8. Publish a post that has no AI readiness score (or one with a
   `ai_readiness_checked_at` older than 24h). After the publish lands,
   a non-blocking toast appears bottom-right: "Just published. Score
   AI readiness now?" with a "Score" CTA. Click "Score" → drawer opens
   and auto-fires a fresh scan against the just-published URL.
9. Disable MCP locally (unset `AI_READINESS_MCP_TOOL_NAME` in
   `.env.local`). Reload an admin page; the button is hidden entirely
   on every post. Re-set the env var to restore.

## 9. Known gaps / open questions

### 9.1 Merge-ordering caveat (binding on TPM)

This branch was developed against M7-BE's working tree. The FE files
on `cms/m7-fe` reference types that only exist post-M7-BE merge:
- `AiReadinessBand`, `AiReadinessCheck`, `AiReadinessCheckStatus`,
  `AiReadinessReport` from `lib/blog-schema.ts`.
- The new `POST_NOT_PUBLISHED` + `NOT_SCORED` error codes.

**Merge order MUST be:**
1. `cms/m7-be` → `cms-dev` first.
2. `cms/m7-fe` → `cms-dev` after.

Reverse order leaves `cms-dev` briefly in a state where the FE
references undefined types. The TPM merge checklist in
`docs/cms/agent-prompts.md` §6 already calls out BE-first merge order
for every milestone — this branch is the same shape.

### 9.2 BE GET handler does not return 503 when MCP is disabled

The dispatch §1 specifies: "Hidden entirely if `post.ai_readiness_band`
is null AND a fetch to `GET /api/admin/posts/[id]/ai-readiness` returns
503 with `MCP_DISABLED`."

The M7-BE GET handler at `app/api/admin/posts/[id]/ai-readiness/route.ts`
does NOT call `isDisabled()` — only POST does. So when MCP is disabled,
GET still returns either 200 (cached) or 404 (NOT_SCORED). The FE's
probe will therefore never receive the 503 hide signal in practice.

**Impact:** the button stays visible in disabled mode; the disabled
state is only revealed when the admin clicks and the drawer's POST
returns 503 (rendering the "Service not configured" banner). Functional
but slightly louder than the dispatch intended.

**Recommendation:** ask M7-BE to add `if (isDisabled()) return
errorResponse(503, 'MCP_DISABLED', …)` at the top of the GET handler,
mirroring POST. One-line change. Filed under §9 rather than blocking
the merge — the FE behaves correctly when the BE returns the documented
shape.

### 9.3 Drawer occupies a fixed 560px right rail, not the preview-pane width

Dispatch §2 says: "drawer takes the preview's width, preview slides
under." The current implementation is a fixed-width (`max-w-[560px]`)
right-side `position:fixed` panel that visually overlays the preview
pane area, but doesn't physically swap into the preview pane's slot in
the editor-shell flex layout.

**Why:** the editor-shell's right-pane width is owned by a draggable
divider + `localStorage` split state. Hooking the drawer into that
plumbing would couple two systems with different lifetimes (drawer
state is editor-form-owned; split state is editor-shell-owned). The
fixed-overlay approach achieves the same visual outcome with zero
coupling and works identically on `<lg` viewports where the editor
falls back to a tab toggle.

**Recommendation:** keep as-is for M7. Revisit if Tester flags the
visual approximation as insufficient; the surrounding editor-shell
layout would need a small refactor to expose a "right rail content"
slot that the drawer could slot into.
