Date: 2026-05-14
Branch: `cms/v2-mongo-be` off `cms-dev`
Owner: Backend agent

This branch consolidates the CMS data layer on MongoDB. No API contracts
change; no FE work is required; no new features are introduced. The team
infra is now centralised on DigitalOcean, where a managed MongoDB cluster
is already provisioned — Postgres was no longer an option.

## 4. Design decisions taken under ambiguity

These are decisions I made where the dispatch left a choice or where the
working tree forced one. Each is reversible — if a future BE agent
disagrees the diff is small.

### 4.1 `_id` strategy (dispatch §5 decision)

Picked option (a) — `_id` is the UUID v4 directly. The wire shape's
`id` field is set in the row→wire mappers (`rowToPost` / `rowToImage`)
from `row._id`. No second `id` field carried in the document. Routes
look up by `{ _id: id }`. Test fixtures read `row._id` where they used
to read `row.id`.

This dropped one field and one source of accidental divergence (no risk
of `_id` and `id` getting out of sync over time).

### 4.2 `db` export is a `Proxy`, not a static binding

`db/client.ts` exports `db` as a `Proxy<Db>` whose every property access
defers to `getDb()`. Two reasons:

- Preserves every existing `import { db } from '@/db/client'` line and
  every `db.collection(...)` call site without renaming.
- Preserves the test-mock seam — route tests already do
  `vi.mock('@/db/client', () => ({ get db() { return testHandle.db } }))`,
  and the proxy means the mock can be a fresh Db handle per test.

The runtime cost is one extra property lookup per call; the savings in
churn outweighed it.

### 4.3 mongodb-memory-server lifecycle

One memory server per Vitest worker (lazy-booted on first
`createTestDb()` call). Each `createTestDb()` returns an isolated Db
backed by a unique random database name on the shared server. The first
boot downloads the MongoDB binary (~70 MB) into the OS cache and takes
~3–10 s; warm subsequent boots are sub-second. I bumped `testTimeout` /
`hookTimeout` to 30 s in `vitest.config.ts` so the first test in each
worker doesn't time out.

I considered top-level await + a single boot at module init, but vitest's
ESM transformer with `target: ES2017` is a fragile combination there.
Lazy-boot is simpler and robust.

### 4.4 Slug uniqueness — race on PATCH

The `PATCH /api/admin/posts/[id]` handler reads the row, optionally
updates the slug, and writes back. Between the read and the write,
another admin could in principle take the slug. The unique index on
`{ slug: 1 }` makes the second write fail with E11000, which the route
catches via `isUniqueViolation` and translates to `422 SLUG_CONFLICT` —
so the user-visible behaviour is correct. No transaction needed.

### 4.5 publish route: `$ifNull` pipeline update

The Postgres design used `COALESCE(published_at, $now)` so re-publish
preserves the original timestamp. The Mongo equivalent is an
aggregation-pipeline update with `$ifNull`:

```js
findOneAndUpdate(
  { _id: id },
  [{ $set: {
      status: 'published',
      published_at: { $ifNull: ['$published_at', now] },
      updated_at: now,
    } }],
  { returnDocument: 'after' },
)
```

Single statement → atomic at the document level. The test
`republish keeps the original published_at` confirms behaviour parity.

### 4.6 Rate-limit count via `countDocuments` not row-scan

The Postgres design selected `id` rows up to `LIMIT cap+1` instead of
using `count(*)::int` because pg-mem returned aggregate results without
column metadata that Drizzle's positional decoder choked on. That
constraint is gone with the native MongoDB driver, so the v2-mongo
rate-limit routes use `coll.countDocuments(filter)` directly. Simpler,
one round-trip, no wasted document materialisation.

### 4.7 TTL indexes replace M8 periodic prune

`login_attempts` and `ai_readiness_attempts` each have a TTL index on
`attempted_at` (3600 s and 7200 s respectively). The Mongo TTL monitor
sweeps the collections automatically every ~60 s. The M7 BE handoff §9.6
noted the count-then-insert race as advisory; the TTL closes the other
half of that gap — neither collection grows unbounded, and the M8
periodic-prune work item is effectively retired.

### 4.8 `app/api/admin/logout/route.test.ts` mock shape

The original mock was `vi.mock('@/db/client', () => ({ db: {}, schema: undefined }))`.
I dropped the `schema: undefined` clause — `@/db/client` no longer
re-exports a `schema` namespace. `db: {}` is still required (route module
load eagerly touches the export even though the handler itself is
DB-free).

### 4.9 BSON Date vs ISO string on the wire

Stored as native BSON `Date`. The row→wire mappers continue to call
`toIso(...)` so the API payloads emit ISO 8601 strings exactly as
before. The existing `isoOf` helper in the ai-readiness route is kept
verbatim.

### 4.10 Connection string parsing

`db/client.ts` parses the database name from the URI path segment so
operators can move the database name into env-var control. Falls back to
`metaborong` if the path is empty. Test code never hits this path —
`createTestDb()` constructs Dbs by name directly.

## 6. Operator notes

### 6.1 Bootstrapping a fresh database

There is no migration step. Indexes are created lazily on first
`getDb()` call per process (idempotent — Mongo treats redundant
`createIndex` calls as no-ops). The four collections are created
automatically on first insert.

To pre-create indexes ahead of the first request, run a one-shot script
that imports `db/client.ts`:

```sh
MONGODB_URI=mongodb://… node -e "
  import('./db/client.ts').then(({ getDb }) => {
    getDb()
    setTimeout(() => process.exit(0), 2000)
  })
"
```

### 6.2 Seeding admin credentials

Unchanged from M1. `ADMIN_EMAIL` is plain text; `ADMIN_PASSWORD_HASH` is
a bcrypt-cost-12 hash generated with:

```sh
node -e "import('bcryptjs').then(m=>m.hash('your-password',12).then(console.log))"
```

Backslash-escape every `$` in the hash before pasting into `.env.local`.
The exact same dotenv-expand caveat applies to `MONGODB_URI` if the
password contains `$`.

### 6.3 Applying the env-var swap to staging / prod

1. Add `MONGODB_URI` to the deployment environment (staging first,
   `www.metaborong.com` second per the CMS staging-vs-prod memo).
2. **Remove `POSTGRES_URL`.** It's unused after this branch lands. Leave
   it in until the first staging deploy completes successfully — it
   doesn't break anything to have an extra env var, but trim it in the
   following pass.
3. The first request after deploy creates the indexes lazily. Watch the
   logs for `[db] ensureIndexes failed:` — if it appears, the deploy
   should be rolled back; the most likely cause is an IP-allowlist
   misconfiguration on the DO managed cluster (see §10).

## 7. Cross-pollination — FE impact

**Should be zero.** API JSON payloads, status codes, and error envelopes
are byte-for-byte identical to the Postgres design. The FE's
`lib/api-client.ts` types (Post, PostSummary, Image, AiReadinessReport,
all error codes) are unchanged. The Zod schemas in `lib/blog-schema.ts`
are untouched.

If a FE test was previously asserting against an `id` field or a
timestamp string, it should still pass — both come through the
unchanged row→wire mappers. I did not encounter any FE callsite that
needed adjustment during this branch.

## 8. Manual verification (golden flow)

```sh
# 1. Login (mints mb_admin_session + mb_csrf)
curl -sS -c /tmp/mb-cookies -X POST http://localhost:3000/api/admin/login \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@metaborong.test","password":"<dev-password>"}'

CSRF=$(awk '/mb_csrf/{print $7}' /tmp/mb-cookies)

# 2. Create draft, capture id
POST_ID=$(curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -H 'Content-Type: application/json' \
  -X POST http://localhost:3000/api/admin/posts \
  -d '{"title":"v2-mongo smoke"}' | jq -r '.post.id')

# 3. Publish it
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -X POST http://localhost:3000/api/admin/posts/$POST_ID/publish

# 4. Hit the public reader (next/cache will revalidate from publish)
curl -sS http://localhost:3000/blog/v2-mongo-smoke/raw.md

# 5. Score AI readiness (requires the M7 env vars set)
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -X POST http://localhost:3000/api/admin/posts/$POST_ID/ai-readiness | jq

# 6. Confirm: cache hit on a second POST is instant + has cached:true
curl -sS -b /tmp/mb-cookies -H "X-CSRF-Token: $CSRF" \
  -X POST http://localhost:3000/api/admin/posts/$POST_ID/ai-readiness | jq '.cached'
```

The `Post` shape returned by every endpoint must look identical to the
Postgres era — same field names, same ISO timestamps, same `id` (which
is now `_id` server-side, transparently mapped to `id` on the wire).

- **9.3 — TTL grace.** The `login_attempts` TTL is 3600 s and the rate

  `ai_readiness_attempts` TTL is 7200 s vs a 3600 s window — 2× margin.
  If either window is ever increased, bump the TTL `expireAfterSeconds`
  accordingly in `db/collections.ts`.
- **9.4 — No equivalent of `pnpm db:migrate`.** Operators who muscle-
  memory that command will need to relearn the lazy-index story. The
  `db/README.md` rewrite makes this explicit but it's worth flagging.
- **9.5 — The `db` Proxy hides type narrowing.** `db.collection<PostDoc>('posts')`
  works fine at the type level, but errors raised from inside the Proxy
  trampoline carry stack frames from the proxy hop. Not a correctness
  issue; just slightly more cryptic stack traces. Acceptable.

## 10. Operational risk — DO Managed MongoDB

**The single biggest deploy-time concern.** Three things to confirm
before flipping staging:

### 10.1 IP allowlist

Vercel serverless functions egress through a rotating pool of IPs in
each region. DigitalOcean Managed MongoDB defaults to a deny-all
allowlist. Two options:

1. **VPC peering.** Vercel does not currently offer VPC peering with DO.
   Cross-cloud peering requires a third-party transit gateway. Not free.
2. **Public access + `0.0.0.0/0` allowlist with TLS + auth.** This is
   how most Vercel↔DO MongoDB setups work in practice. The wire is
   TLS-encrypted, auth is bearer-style, and unbounded source IPs is
   safe as long as the connection string secret stays out of the repo.
   Confirm with the security review before flipping `0.0.0.0/0`.

My recommendation, based on a quick read of DO's docs: option 2 with
`tls=true&authSource=admin` in the URI, and rotate the password if
anyone ever sees `MONGODB_URI` in plaintext.

### 10.2 SRV vs standard URI

DO Managed MongoDB historically required the standard (`mongodb://...`)
form, not the SRV (`mongodb+srv://...`) form. The driver supports both,
but DNS configuration must match. **Confirm at deploy time which form
the DO console returns** — copy verbatim into `MONGODB_URI`.

### 10.3 Connection pool sizing on serverless

The default MongoClient pool is 100 connections. On Vercel Hobby with
many concurrent function instances, that's far more than the DO cluster
can handle (DO's smallest instance caps at ~150 concurrent connections
*total*). Two mitigations:

1. Set `maxPoolSize=10` in the URI options.
2. The current `db/client.ts` already caches the client on `globalThis`,
   which means warm-starts reuse the pool — only cold starts open new
   ones. This combined with `maxPoolSize=10` should fit comfortably.

If a deploy starts seeing connection-saturation errors, this is the
first place to look.

### 10.4 Semantic losslessness check

We never used any Postgres-specific feature beyond UUID PKs, JSONB,
GIN-on-array, and unique constraints. No stored procedures, no triggers,
no full-text search, no PostGIS, no LATERAL joins. Every read/write the
Postgres design did has a 1:1 MongoDB equivalent in this branch.

**Migration is semantically lossless.** This is worth flagging for
confidence — the deploy-time risks are operational (IP allowlist,
connection pool sizing) not behavioural.
