# MCP BE — Claude MCP server for CMS authoring (handoff)

Date: 2026-05-15
Branch: `cms/mcp-be` off `cms-dev @ 807ce19`
Owner: Backend agent

`POST /api/mcp` exposes the CMS write surface to Claude (and any MCP-aware
client) via JSON-RPC 2.0 / "Streamable HTTP" (2025-03-26). v1 is
**draft-only on purpose** — Claude can create / patch drafts and upload
images, but cannot publish, unpublish, or delete. Publishing remains a
deliberate human action in `/admin/blog`.

No API contract change for the existing admin surface. No FE work
required. The existing 573-test suite continues to pass; this branch
adds 58 new tests (total 631).

| `cms_list_posts`    | List posts (filter+paginate)                                               | `{ status?: 'draft' \| 'published', limit?, offset? }`                                                                                          | `{ posts: PostSummary[], total }`                   | `VALIDATION_FAILED`                                                                                                 |

JSON-RPC error codes live in `lib/mcp/types.ts`. The `error.data.code`
field carries the CMS-side machine code (mirrors `/api/admin/**`) so a
client built against the admin error envelopes can correlate without
new code.

## 4. Design decisions taken under ambiguity

### 4.1 Admonition extraction — post-AST walker, not a remark plugin

The dispatch (§4) left the choice between a custom remark plugin or a
post-AST walker. **Picked the walker.** Reasoning: a remark plugin would
require us to declare a `visitor` over generic nodes, but the marker we
care about (`[!tip]`, `[!takeaway]`, `[!faq]`) lives in the rendered
plain-text of the blockquote's first paragraph, not in the AST shape
itself. A plugin would have to invent a new node type or rewrite the
blockquote inline; both are heavier than reading `blockquote.children[0]`
in the walker. The walker also keeps the entire conversion in one file
so future maintainers don't have to chase down a separate plugin module.

### 4.2 Role-hint precedence

Two mechanisms (HTML comment `<!-- role: tldr -->` and `block_roles`
map). **Map wins on conflict** at the same index. This is explicit in
the dispatch §6, but I want to flag it again: the map is the
machine-readable signal Claude is most likely to use, the comment is the
human-readable fallback. If Claude sets both at the same index, the
machine signal is the source of truth.

### 4.3 H1 in markdown — hard rejection, not silent demotion

The dispatch §6 says "H1 in markdown → reject with INVALID_CONTENT". I
considered silently demoting to H2 (less friction for Claude) but
rejected: the post title lives on its own field, an H1 in the body is
almost always a content-shape mistake, and surfacing it as an error
gives Claude a clear correction path. Logged here so we don't redebate.

### 4.4 Image src — UUID-only, no URL passthrough

`![alt](https://example.com/x.jpg)` is **rejected**, even though
CommonMark accepts arbitrary URLs. The trade-off: we lose the ability
to draft posts that reference external images, but we keep the post
schema's `image.imageId: uuid` invariant intact. Claude has to call
`cms_upload_image` first and use the returned UUID — same workflow the
admin uploader enforces.

### 4.5 Inline markdown — stripped to plain text

Bold (`**`), italic (`*`), inline code (` ` `), links (`[…](…)`) all
collapse to their visible text. The FE renderer
(`lib/editor/serialize.ts`) treats `block.data.text` as plain text, so
preserving the markdown syntax here would surface raw asterisks in the
final HTML. Rich inline support is a v2 backlog item; flagged in §9.

### 4.6 `cms_list_posts` — fetch-all-then-slice

We do not currently paginate at the Mongo layer. `listAllPostsForAdmin`
returns the whole catalog and the tool slices `[offset, offset + limit]`.
The dashboard already does this and the post count is small enough
(low hundreds) that the round-trip cost is negligible. If the catalog
grows past ~10k posts, swap this for a real cursor-based query — but
that's a bridge to cross then, not now.

### 4.7 JSON-RPC error envelope — extra `data.code` field

JSON-RPC 2.0 only requires `code` (integer) + `message` on errors. We
additionally surface `error.data.code` (CMS machine code) and
`error.data.field` (dotted path). This matches what `/api/admin/**`
emits in its plain-HTTP envelope, so a client built against either
surface can use the same error-mapping table.

### 4.8 `tools/call` response shape

MCP standard wraps tool results in a `content: [{ type, text }]` array.
We emit `content: [{ type: 'json', json: result }]` (using the `json`
content kind instead of `text` so clients don't have to JSON.parse the
string). For convenience we ALSO expose the raw result at the top level
as `payload`. The MCP spec allows extra fields and most clients read
both; if a future client breaks on the `payload` key, drop it without
churn.

## 6. Operator notes

### 6.1 Minting `MCP_ADMIN_TOKEN`

```sh
openssl rand -base64 32
```

Set in `.env.local` for local dev:

```
MCP_ADMIN_TOKEN=<output>
```

Remember the dotenv-expand caveat: backslash-escape any `$` in the
token value (`$` → `\$`). Without escaping, Next strips the `$x` segment.

### 6.2 Vercel env

```sh
vercel env add MCP_ADMIN_TOKEN production
vercel env add MCP_ADMIN_TOKEN preview      # mint a SEPARATE token for preview
```

The CMS already targets `staging` (per the standing memo); production
cutover for www.metaborong.com is a TPM-owned step. **Use distinct
tokens per environment** so a leaked preview token can't be replayed
against production.

### 6.3 Claude Code config

Add the server to `~/.config/claude-code/config.json` or, for
project-scoped use, `.claude/settings.json`:

```json
{
  "mcpServers": {
    "metaborong-cms": {
      "type": "http",
      "url": "http://localhost:3000/api/mcp",
      "headers": {
        "Authorization": "Bearer <MCP_ADMIN_TOKEN>"
      }
    }
  }
}
```

For staging, swap the URL to `https://<staging-domain>/api/mcp`. Same
shape for `claude.ai` MCP integrations (Settings → MCP Servers).

## 8. How to manually verify (curl recipe)

```sh
TOKEN="<your-MCP_ADMIN_TOKEN>"

# 1. Handshake
curl -s http://localhost:3000/api/mcp \
  -H "content-type: application/json" \
  -H "authorization: Bearer $TOKEN" \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize"}' | jq

# 2. List tools
curl -s http://localhost:3000/api/mcp \
  -H "content-type: application/json" \
  -H "authorization: Bearer $TOKEN" \
  -d '{"jsonrpc":"2.0","id":2,"method":"tools/list"}' | jq

# 3. Create a draft
curl -s http://localhost:3000/api/mcp \
  -H "content-type: application/json" \
  -H "authorization: Bearer $TOKEN" \
  -d '{
    "jsonrpc":"2.0","id":3,"method":"tools/call",
    "params":{
      "name":"cms_create_draft",
      "arguments":{
        "title":"Hello from curl",
        "markdown":"## A heading\n\nA paragraph.\n\n> [!tip] try the MCP tool"
      }
    }
  }' | jq

# 4. Reject path: bad bearer
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3000/api/mcp \
  -H "content-type: application/json" \
  -H "authorization: Bearer wrong" \
  -d '{"jsonrpc":"2.0","id":1,"method":"initialize"}'
# → 401
```

## 9. Known gaps (v2 backlog)

- **Rich inline markdown not surfaced yet.** Bold/italic/links collapse
  to plain text. The FE renderer treats `block.data.text` as plain.
  Adding rich-inline support requires a tiptap-side schema change too,
  so it's bundled into a future FE+BE slice.
- **No publish tool.** Intentional — slop-prevention. If we ever surface
  one, gate it on a separate `MCP_PUBLISH_TOKEN` env var with a stricter
  rate limit, and emit a notification (Slack / email) on every use.
- **No delete tool.** Same reasoning. Admin UI only.
- **No batch image upload.** One image per call. A multi-image call
  would be a small extension if Claude usage patterns warrant it.
- **No webhook back to Claude on publish.** If a human publishes a
  draft Claude authored, Claude is not notified. Out of scope for v1.
- **`cms_list_images` not exposed.** Claude must always
  `cms_upload_image` to get an id; can't pick an existing one. Adds one
  more tool; weight cost vs convenience — flagged for the TPM as a §10
  open question.
- **HTTP+SSE transport not supported.** v1 is Streamable HTTP only.
  Older MCP clients that expect SSE will need to upgrade. Documented
  in the operator notes; if usage demands it we can wrap.

## 10. Operational risk

### 10.1 Vercel timeout on large image fetches

`cms_upload_image` with a URL source can wait up to 10s for the upstream
fetch. Vercel's Hobby plan has a 10s function timeout, so an
`upstream-slow` fetch + the sharp transcode + the blob upload could
breach the cap. Mitigation: the explicit 10s `AbortController` keeps
fetches bounded; sharp transcode is sub-second for our 8 MB cap; blob
upload is the wildcard. **If you see Vercel function timeouts under
real load, the fetch + transcode + upload should be moved to a
background task** (post a "pending" record, finish out-of-band, surface
status via a follow-up `cms_get_image` tool).

### 10.2 Race condition: two clients calling `cms_create_draft` with the same slug

Mongo's unique index on `posts.slug` makes the write race-safe — the
second writer gets E11000 → `SLUG_CONFLICT`. The dispatch §13 flagged
this; behaviour is verified by `cms_create_draft > SLUG_CONFLICT on
duplicate slug` in `lib/mcp/tools.test.ts`.

### 10.3 Bearer token leak

The token is admin-equivalent. Anyone with it can create drafts that
shape an editor's eventual review queue. Mitigations:
- Use **distinct tokens per environment** (preview vs staging vs prod).
- Rotate via `vercel env rm` + re-mint when in doubt.
- The route does NOT log the token on any path (verified by code
  inspection — no `console.log` near the auth check).

### 10.4 Storage growth from abandoned Claude-authored drafts

Drafts created via MCP never auto-publish, but they also never
auto-expire. If Claude churns through hundreds of attempts, drafts
accumulate. v1 acceptable (the admin UI can bulk-delete from the
dashboard); v2 should add a `last_touched_by` field and a sweeper for
drafts older than 30 days with `last_touched_by = 'mcp'`.
