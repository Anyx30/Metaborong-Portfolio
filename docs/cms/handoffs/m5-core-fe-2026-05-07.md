## 3. Components created (one-line each)

- `TableOfContents` — Client Component. Renders nothing for posts with
  zero headings (PRD edge case). Anchor click smooth-scrolls and
  updates `window.location.hash` via `history.pushState`. Active
  section highlighted via `IntersectionObserver` (rootMargin tuned so
  the active row stays meaningful at the top/bottom of the article).
  Two variants:
  - `variant="desktop"` — sticky right-rail (`top: 100px`, scroll-on-overflow),
    indented per heading depth, `aria-current="location"` on the active row.
  - `variant="mobile"` — `<details>` accordion with an `On this page`
    summary; same anchors, same a11y attributes.
- `extractHeadings(blocks)` — pure helper exported alongside the
  component so the component test (and any future renderer) can verify
  the extraction in isolation.

### Playwright E2E

`e2e/m5-core-seo.spec.ts` — 4-test serial suite (gated on
`E2E_ADMIN_EMAIL` + `E2E_ADMIN_PASSWORD`):

1. **Seed + publish a post with headings.** Login → New post → fill
   title → land on edit. PATCH `content_json` (3 heading blocks +
   2 paragraphs) + author + excerpt + tags directly via the admin API
   (same pattern as M2/M4 specs). Reload editor; click Publish; assert
   `Status: Published`.
2. **Public `/blog/[slug]` inlines Article + BreadcrumbList JSON-LD.**
   Incognito context. Retries up to 5x for ISR propagation. Reads
   every `<script type="application/ld+json">`, parses each, asserts
   `@type` set contains `Article` AND `BreadcrumbList`, asserts the
   Article has a `headline`, `datePublished`, and an `image` matching
   `/og?slug=…`. Asserts the BreadcrumbList has 3 `itemListElement`s
   with the post's title at position 3.
3. **TOC anchor click updates URL hash.** Forces a 1440×900 viewport
   so the desktop TOC variant renders. Locates the navigation
   landmark by accessible name, clicks the "How we tested it" link,
   waits for `window.location.hash === '#how-we-tested-it'`.
4. **`/blog/rss.xml` returns valid RSS 2.0 with the post listed.**
   Asserts 200, `application/rss+xml` content-type, opens with
   `<?xml`, contains `<rss version="2.0"`, `<channel>`, the seeded
   post title (XML-escaped), and the `/blog/<slug>/` URL.

`afterAll` re-logs-in and DELETEs the seeded post via API, even on
mid-suite failure, so the dev DB doesn't leak rows between runs.

## 8. Manual-verify path (5-step golden flow)

Assumes the M2 + M3 surfaces are merged into `cms-dev` (or this
branch's predecessor cherry-picks have applied). Postgres on `:5433`,
dev server on `:3000`, admin creds seeded
(`admin@metaborong.test` / `admin1234`).

1. **Seed a post with headings.** Login at `/admin/login` →
   New post. Title `M5 smoke`. Open the editor, type a few headings
   (`Section A` h2, `Subsection` h3, `Section B` h2) and a paragraph.
   Cmd+S. Publish.
2. **Public render.** Visit `/blog/m5-smoke` in an incognito window.
   View source. You should see TWO `<script type="application/ld+json">`
   tags directly inside `<body>`. Pretty-print one — it's the Article
   schema with `headline`, `datePublished`, `image`, `author`,
   `publisher: Metaborong`. The other is the BreadcrumbList with
   Home → Blog → Post.
3. **TOC.** On a wide viewport (≥ 1024px) you should see a sticky
   "On this page" rail on the right with `Section A`, `Subsection`,
   `Section B`. Clicking `Subsection` smooth-scrolls and the URL
   becomes `…/m5-smoke#subsection`. Resize the window below 1024px:
   the rail collapses into a `<details>` accordion above the article
   with the same links. Set the post to have NO heading blocks; the
   TOC disappears entirely.
4. **OG card.** `curl -I http://localhost:3000/og?slug=m5-smoke` returns
   200 + `image/png`. `curl -s …/og?slug=m5-smoke -o /tmp/og.png &&
   open /tmp/og.png` shows a 1200×630 card with the post title, excerpt
   tagline, and tag pills.
5. **RSS.** Visit `/blog`. Top of the listing has an RSS link beside
   the optional "← All posts" tag-filter link. Click it →
   `/blog/rss.xml`. The browser renders the XML directly. The post
   you just published is in the feed.

## 9. Known gaps / open questions

1. **`articleSchema` signature is `({ post, imageUrl })` not `(post, region, options)`.** The BE
   handoff §7.3 documents the same drift. Both forms cover the v1
   surface; the FE call site `articleSchema({ post })` and the test
   suite both validate the object-arg form. If a future region-aware
   tweak (e.g. `inLanguage`) is needed, this becomes a contract change
   touching `lib/seo.ts` + every call site.
2. **Cover-image URL is not yet plumbed into JSON-LD `image`.** Every
   post defaults to `/og?slug=…` for the share card. M4's
   `getImageById` would resolve a real cover URL — wiring that into
   `articleSchema({ post, imageUrl: cover?.blob_url })` is a small
   follow-up. Doesn't block M5-core because the OG fallback is
   well-formed and unique per post.
3. **`<description>` missing on RSS items.** The cherry-picked
   `76c2442` BE commit does not emit `<description>` on items. BE
   handoff §7.1 flags this as a defect. FE consumes the route as-is;
   when BE re-dispatches with `listPublishedPostsForFeed` the FE
   surface (the RSS link) doesn't change.
4. **TOC IntersectionObserver thresholds are tuned for typical
   article lengths.** Short articles where multiple h2s fit in one
   viewport pass may briefly highlight the wrong row on the upward
   scroll edge. This is cosmetic — anchor clicks always set the
   correct active row immediately.
5. **`history.pushState` adds an entry per anchor click.** Hitting
   browser-back after a few TOC clicks walks back through them.
   Acceptable for v1; if the editorial team finds it noisy we can
   swap to `history.replaceState`.
6. **E2E spec not executed in this dispatch.** Same caveat as M3/M4.
   The spec compiles; Tester should run it under their seeded admin
   DB.
7. **The session hook described in §1 reverted edits to
   `vitest.config.ts`, `app/sitemap.ts`, etc., between checkouts.**
   None of those files are FE-owned for M5-core. The committed BE
   branch is the canonical state.
