## 4. Item-by-item notes

### 4.1 Item 1 — orphan `block_overrides[id]` GC

Spec: M9-C Tester report L1.

`pruneVariants` gained an optional `liveBlockIds?: Set<string>` parameter. When supplied, the override-map iteration drops entries whose `blockId` is not in the set; when omitted, the existing behavior holds (every override survives the prune). Two call sites pass the set:

1. `runSave` derives `liveBlockIds` from `validation.parsedContent` (the schema-validated Block[]) so the PATCH body's `geo_variants` never carries a dead override.
2. The `dirty` memo derives `liveBlockIds` from the live `blocks` state so deleting an overridden block flips `dirty=true` immediately, not just when the next autosave's prune lands.

Both call sites use the same `new Set(b.id for b in blocks)` shape; no helper extracted (one-liner, two sites).

Skipping the filter when `liveBlockIds` is undefined preserves the existing behavior on any future autosave-pre-validation path that doesn't have a parsed `Block[]` yet — defensive, no current callers omit the set.

### 4.2 Item 2 — drawer 503 banner copy

Spec: M7 Tester report N1.

`drawer.tsx:212` previously rendered `"Set the AI_READINESS_MCP_* env vars in your deployment and reload."` — admin-facing operator hint, but the literal `AI_READINESS_MCP_*` substring shipped the env-var prefix into a client-rendered string. Replaced with the dispatch's prescribed copy: `"AI Readiness service isn't configured. Ask the site admin to enable it."`

The existing 503 vitest case retains its `/service not configured/i` assertion and gains two negatives: `not.toMatch(/AI_READINESS_/i)` and `not.toMatch(/MCP_/)`. Both regexes are belt-and-braces — the second catches any future lapse that names a different MCP_* env var.

The BE error envelope (`{ error: "AI Readiness service is not configured.", code: "MCP_DISABLED" }` from `lib/api.ts:errorResponse`) was already neutral and stays unchanged. The client-side copy is now the only banner string the admin reads, and it carries no env-var prefix.

### 4.3 Item 3 — Retry-After test fake-timer pin

Spec: M7 Tester report N2.

The pre-existing 429 case asserted only `/Try again at \d{2}:\d{2}/` — an off-by-one in the seconds-vs-ms math (or in the `* 1000` factor) would pass undetected. The new case pins `vi.setSystemTime(new Date('2026-05-08T14:00:00Z'))` and mocks a 429 with `retryAfter: 1800` (30 minutes), then asserts the rendered banner contains the exact HH:MM derived the same way the drawer derives it (`new Date(Date.now() + 1800 * 1000)` followed by `.getHours()` / `.getMinutes()` in local TZ). The expected pair is computed in the test rather than hardcoded, so the assertion is timezone-agnostic and stays green on any runner.

`vi.useFakeTimers({ shouldAdvanceTime: true })` is required — Testing Library's `findByTestId` polls via `setTimeout`, which would stall under the strict fake-timer mode and time the suite out (caught on the first run; fixed by switching the fake-timer flavor to match the SaveState lifecycle test in `edit-post-form.test.tsx:209`). The `try { ... } finally { vi.useRealTimers() }` wrapper guarantees the test doesn't leak timer state into subsequent cases in the same file.

The drawer's production rate-limited path (`drawer.tsx:192-203`) is unchanged — Item 3 is a test-only addition.

### 4.4 Item 4 — `autoScanFiredRef` latch test

Spec: M7 Tester report N3.

The dispatch wording — *"close+reopen … assert fetch DOES fire (latch is reset on close, or scan fires on each fresh open — whichever the current behavior is; codify it)"* — turned out to need careful reading. The actual behavior, by code-walk on `drawer.tsx:104-118`:

- Mount with `open=true`, `initialReport=null`, state starts as `{kind: 'idle'}`.
- First effect: `open && !latch && state.kind !== 'ready' && !initialReport` → POST fires, latch flips to `true`. State transitions `'loading'` → `'ready'`.
- Same-prop re-render: latch is `true` AND state is `'ready'` → effect bails twice over. **No second POST.** ✓ (this is the dispatch's primary assertion)
- `open=false` → component returns `null` at the top level (`drawer.tsx:131`) but the React instance is *not* unmounted (parent still renders `<AiReadinessDrawer>` in JSX). The second effect (`drawer.tsx:116-118`) resets `autoScanFiredRef.current = false`. State is **not** reset — local `useState` survives.
- `open=true` again on the same instance: latch is `false` (reset above) BUT `state.kind === 'ready'` → first effect bails on the state guard. **No second POST.**

So "the latch resets on close, but the cached state persists, so reopen stays a no-op" is the actual current behavior. This matches the M7 e2e cache-hit case 4 (`Re-open does not re-POST`) and is what the new vitest case codifies — a single test with three phases (initial mount, same-prop re-render, close→reopen) all asserting `apiPost.mock.calls.length === 1`.

The soft-prompt's "Score" CTA exercises a different path — fresh component mount with `initialReport=null` because `aiDrawerSeedNull` flips the prop — already covered by the existing `auto-fires a POST when opened without an initialReport` case at `drawer.test.tsx:208`. Documented inline in the new case so a future reader doesn't conflate the two paths.

### 4.5 Item 5 — `mb.editor.variant.<id>` localStorage cleanup

Spec: M9-C handoff §9 v1.6 item 5.

Two delete sites — the edit-post-form's modal (`edit-post-form.tsx:489`) and the dashboard list's `DeleteConfirmModal` (`admin-posts-list.tsx:302`) — both gain a single `try { window.localStorage.removeItem('mb.editor.variant.' + <id>) } catch {}` line after the `api.delete` resolves. Silent-swallow matches the existing pattern at `edit-post-form.tsx:218-231` (load + persist), where private-mode storage throws are absorbed quietly.

The dashboard-list path has no separate test — its 10 existing cases stay green; the call-site change is a literal one-line copy of the form's pattern and the same assertion would not catch any failure mode the form's test doesn't.

The dispatch's *optional bonus* (sweep all `mb.editor.variant.*` keys on `/admin` mount) was **not** taken.
