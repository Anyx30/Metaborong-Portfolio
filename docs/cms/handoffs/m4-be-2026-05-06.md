## 3. New env vars

| Var                       | Purpose                                                | Consumed by                                |
|---------------------------|--------------------------------------------------------|--------------------------------------------|
| `BLOB_READ_WRITE_TOKEN`   | Auth for `@vercel/blob` `put` / `del`                  | `lib/images.ts` (via `@vercel/blob`)       |

Provisioning steps for ops:
1. Vercel Dashboard → Storage → Blob → "Connect Project" → Generate Token.
2. Copy the value.
3. Paste into `.env.local`. **Backslash-escape every `$`** in the token.
   Vercel Blob tokens often contain `$` separators, and Next.js loads
   `.env.local` through `dotenv-expand` which would silently substitute
   any unescaped `$<name>` segment with the empty string — exactly the
   same trap M1 documented for `ADMIN_PASSWORD_HASH`. Caught the hard
   way during M1 manual testing; not making the same mistake twice.

The token is server-side only — the Frontend never reads it.

## 5. Edge cases handled

Mapped to "EDGE CASES YOU OWN (M4 binding)" from the dispatch:

| Edge case                                                                                              | Status |
|--------------------------------------------------------------------------------------------------------|--------|
| 415 `UPLOAD_BAD_TYPE` for bad MIME / magic-byte mismatch                                               | PASS — `sniffImageFormat()` checks the leading 12 bytes (JPEG `FF D8 FF`, PNG `89 50 4E 47 0D 0A 1A 0A`, WebP `RIFF…WEBP`); anything else → 415. Sharp transcode failure also collapses to 415 so a corrupted-but-magic-matching buffer doesn't 500. Verified by `rejects a GIF disguised as JPEG`. |
| 413 `UPLOAD_TOO_LARGE` for oversized                                                                   | PASS — three layers: (1) honest Content-Length > 8 MB + 64 KB → 413 immediately, (2) bounded `readBoundedBody()` aborts at 8 MB + 64 KB and returns null → 413, (3) post-parse `file.size > 8 MB` → 413. Verified by `rejects a 9 MB payload`. |
| 422 on missing required field on PATCH (`focal_x` out of [0,1])                                        | PASS — `patchImageBodySchema` Zod schema catches range, with `field='focal_x'`/`'focal_y'` populated from `issue.path`. Verified by `returns 422 for focal_x > 1` and `returns 422 for negative focal_y`. Boundary values 0 and 1 are accepted (verified). |
| All non-GET routes verify CSRF                                                                         | PASS — POST and PATCH both call `requireCsrf(req)` first, before `requireAdmin(req)` and before any DB read. Verified by `403 without CSRF header` on POST and `403 without CSRF` on PATCH. |
| 401 `UNAUTHORIZED` if no session                                                                       | PASS — `requireAdmin(req)` short-circuits on missing or invalid session. Verified by `401 without a session` on GET / POST / PATCH. |
| Stream the multipart body — do NOT buffer before validating size                                      | PASS — `readBoundedBody()` reads `req.body` via `getReader()` and aborts at the first byte beyond the cap; never buffers more than `IMAGE_MAX_BYTES + 64 KB`. The reconstructed Request is what gets passed to `formData()` for parsing, not the live request. |
| 502 + `INTERNAL` + server-side log on Vercel Blob failure                                              | PASS — wrapped `uploadWebpToBlob()` call in try/catch, logs the raw error with `console.error`, returns 502 with code `INTERNAL` and the user-facing message `'upload failed'`. (502 chosen because the upstream is at fault; per §2.2 we don't have an `UPLOAD_FAILED` code, so `INTERNAL` is the right slot.) |
| DB insert AFTER blob upload, with orphan-cleanup on insert failure                                     | PASS — INSERT is wrapped in try/catch; on failure (or no row returned) we `void bestEffortDeleteBlob(uploaded.url)` and return 500 `INTERNAL`. Cleanup is fire-and-forget so it doesn't block the response; cleanup errors are logged at warn but swallowed. |
| `focal_x` / `focal_y` ∈ [0, 1]                                                                          | PASS — Zod `z.number().min(0).max(1)`; tested at boundaries 0 / 1 (accepted) and 1.5 / -0.1 (rejected). |
| `alt` is a string, empty allowed                                                                       | PASS — Zod `z.string().max(2000)`; empty string is valid at this layer. The Frontend will gate block-level alt-required separately (per §2.5 "image block: alt required, non-empty"). |
| GET pagination: cursor is opaque base64-encoded `(created_at, id)`, page size 24                        | PASS — `encodeCursor()` / `decodeCursor()` use base64url; SQL where clause is `(created_at < $1) OR (created_at = $1 AND id < $2)` keyset-style. Verified by `paginates 30 images into a 24/6 split using nextCursor` (no overlap, total = 30, newest-first). Malformed cursor falls through to first page. |
| Random server-generated filename — never round-trip user filename                                     | PASS — `uploadWebpToBlob()` always writes to `images/<crypto.randomUUID()>.webp`; user filename is `sanitizeOriginalFilename`'d and stored in `images.filename` for display only. Verified by `does not round-trip the user-supplied filename into the blob path` (input `'../etc/passwd.jpg'` → blob URL contains neither `passwd` nor `etc`). |
| EXIF stripped on output                                                                                | PASS — sharp's default behaviour drops all metadata unless `.withMetadata()` / `.keepMetadata()` is called explicitly. `.rotate()` (no args) applies EXIF orientation and removes the tag. Verified indirectly by `transcodes a PNG input to a WebP buffer (no EXIF on output)` — output `metadata().exif` is undefined. |
| Errors normalise to §2.2 envelope                                                                      | PASS — every non-2xx response goes through `errorResponse()` from `lib/api.ts` (`{ error, code, field? }` with optional `Retry-After`). |

### Bootstrap a session (re-used by every recipe below)

```sh
curl -i -c /tmp/mb.txt \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@metaborong.com","password":"hunter2"}' \
  http://localhost:3000/api/admin/login
# expect: 200 + Set-Cookie: mb_admin_session, mb_csrf

CSRF=$(awk '/mb_csrf/ {print $7}' /tmp/mb.txt)
```

### Happy upload (multipart) → 201

Save any small JPEG to `/tmp/cat.jpg` first. Then:

```sh
curl -i -b /tmp/mb.txt \
  -H "X-CSRF-Token: $CSRF" \
  -F "file=@/tmp/cat.jpg;type=image/jpeg" \
  http://localhost:3000/api/admin/images
# expect: 201
# body:   { "image": { "id": "...", "blob_url": "https://<store>.public.blob.vercel-storage.com/images/<uuid>.webp",
#                       "width": …, "height": …, "alt": "", "focal_x": 0.5, "focal_y": 0.5,
#                       "filename": "cat.jpg", "created_at": "..." } }
```

The blob URL is `images/<server-uuid>.webp` — your filename never
appears in the path.

### Disguised GIF (Content-Type lies) → 415

```sh
echo -ne 'GIF89a\x01\x00\x01\x00\x80\x00\x00\x00\x00\x00\xFF\xFF\xFF' > /tmp/fake.jpg

curl -i -b /tmp/mb.txt \
  -H "X-CSRF-Token: $CSRF" \
  -F "file=@/tmp/fake.jpg;type=image/jpeg" \
  http://localhost:3000/api/admin/images
# expect: 415
# body:   { "error": "Only JPG/PNG/WebP allowed.", "code": "UPLOAD_BAD_TYPE" }
```

### Oversized payload → 413

```sh
dd if=/dev/urandom of=/tmp/big.bin bs=1M count=9 2>/dev/null

curl -i -b /tmp/mb.txt \
  -H "X-CSRF-Token: $CSRF" \
  -F "file=@/tmp/big.bin;type=image/jpeg" \

# body:   { "error": "file exceeds 8MB limit", "code": "UPLOAD_TOO_LARGE" }
```

### List images (first page) → 200


## 8. Known gaps / open questions

- **Image delete is deferred to v1.5+.** Per the dispatch, M4 explicitly
  ships without `DELETE /api/admin/images/[id]`. Images are independent
  reusable assets — the M2 `DELETE /api/admin/posts/[id]` already does
  not cascade-delete them. Admins can edit `alt` / focal but cannot
  remove an image in v1. When this lands, it will need to: (a) hard-
  delete the DB row, (b) `del()` the Vercel Blob URL, (c) null any
  `posts.cover_image_id` / `posts.og_image_id` that reference it (the FK
  is already `ON DELETE SET NULL` from M1), (d) optionally scan
  `posts.content_json` for `image` blocks referencing this id and warn
  the admin / null the references inline.

- **No content-hash deduplication.** If an admin uploads the same JPEG
  twice, we get two `images` rows and two blobs. Adding a SHA256-based
  dedup is straightforward (hash the WebP output, look up an existing
  row, return it instead of inserting) but out of M4 scope. Track for
  M8 hardening if storage costs become a concern.

- **No admin-level rate limit on POST.** The login route is rate-limited
  per-IP via `login_attempts`; image upload uses the same admin session
  but doesn't have its own counter. A malicious-or-confused admin could
  spam uploads. The same pattern (Postgres-backed counter table) used
  for login can be reused — flag for M8 hardening.

- **`addRandomSuffix: false` relies on `crypto.randomUUID()` for path
  uniqueness.** Collision probability is negligible for v1 traffic but
  not zero. If we ever see one, the INSERT fails on a non-unique blob URL
  (the `images.blob_url` column is currently *not* `UNIQUE` — INSERT
  succeeds, two rows point to the same URL, the older one gets
  effectively replaced when the second `put()` runs). Adding a UNIQUE
  index on `blob_url` (or switching to `addRandomSuffix: true`) is a
  safer posture; not blocking for v1.

- **`@vercel/postgres` deprecation continues to apply** (carried forward
  from M1). Not addressed in this dispatch.

- **Sharp's WebP encoder has a per-image-pixel ceiling.** Anything close
  to that ceiling will throw inside `transcodeToWebp()`, which we
  collapse to 415 `UPLOAD_BAD_TYPE`. This is the right user-facing
  behaviour (admin sees a "bad type" message and re-tries with a
  resized image) but the response message is mildly inaccurate. A more
  honest message ("image too large to process") would require a
  separate code; deferred to UX polish.

- **Multipart parser via Web Request.** We re-construct a `new Request`
  from the bounded body and call `.formData()` on it. This works
  because the Web `Request` constructor preserves the original
  `content-type` (with boundary) when we pass `headers: req.headers`.
  If a future Next.js / undici change alters this, we'd need to swap to
  a multipart parser library. The current approach has zero new deps
  and passes all 12 route tests today.
