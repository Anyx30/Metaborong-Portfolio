  from the DOM, so the `document.body.contains(trigger)` gate makes
  the restore a no-op (focus falls to the natural document order, not
  to a stale element).
- `components/admin/images/image-picker.tsx` — same `triggerRef`
  focus-restore pattern applied to the image picker. The existing
  "focus the close button on open" useEffect was extended to also

## 7. Lighthouse run procedure

Live execution from this dispatch was not attempted because the
session-wide hook described in §1 makes a long-running dev server
unsafe (every working-tree switch could move the route's source mid-
profile). The procedure below produces a deterministic measurement
when run by Tester or in a CI-like environment.

### 7.1 One-time setup
```
pnpm install
# Make sure the BE branch's migrations are applied to your local
# Postgres (mb-test-pg on :5433 per prior Tester sessions).
pnpm next build         # fresh prod-mode bundle so Lighthouse
                        # measures the optimised CSS/JS, not Turbopack dev.
PORT=3000 pnpm next start &
```

Wait for `Ready` on stdout. Confirm `curl -I localhost:3000/blog` returns
200.

### 7.2 Pick a representative published post
A "representative" post for v1 is one with: a cover image, ≥3
heading blocks (so the TOC renders), one inline image block, one
FAQ block, and tags. The M4 / M5-core Tester reports both used
`/blog/m4-smoke-test` or `/blog/m5-smoke` as fixtures — pick the
most recent or seed one via the admin shell.

### 7.3 Run Lighthouse (mobile profile)
Two equivalent options. Prefer (a) so the dispatch's exact
representative-post URL is used.

**(a) Chrome DevTools (manual):**
1. Open the chosen `/blog/[slug]/` URL in Chrome incognito.
2. DevTools → Lighthouse tab → "Mobile" + "Navigation".
3. Categories: Performance, Accessibility, Best Practices, SEO.
4. Click "Analyze page load". Record the four scores.

**(b) Lighthouse CLI:**
```
npx -y lighthouse "http://localhost:3000/blog/<slug>/" \
  --preset=desktop=false --form-factor=mobile \
  --only-categories=performance,accessibility,best-practices,seo \
  --chrome-flags="--headless=new --no-sandbox" \
  --output=html --output-path=/tmp/lh-blog-slug.html
```

### 7.4 Pass criteria (from dispatch)
- SEO ≥ 100
- Performance ≥ 90 (mobile)
- Accessibility ≥ 95
- Best Practices ≥ 90

### 7.5 Likely blockers & how to read them

The static review identified one expected medium blocker that is NOT
fixed in this dispatch and one expected low. Tester should expect to
see them in the Lighthouse report:

1. **Render-blocking fonts** (Performance: probably -3 to -8 points).
   `app/globals.css` has three `@import url(…fontshare/google fonts…)`
   declarations at the top of the file. CSS `@import` is render-
   blocking AND serial — each download blocks the next. Fix is `next/font`
   self-hosting (Satoshi via fontshare, JetBrains Mono and Space
   Grotesk via Google Fonts). **Out of M8-core scope** because it
   touches the homepage's font layer (`app/page.tsx`,
   `components/sections/*`, `components/hero-orb/*`) which the FE
   dispatch carves out as off-limits. Documented in §8.

2. **Cover image LCP** (Performance: probably 0 to -2 points). The
   cover `<Image>` already carries `priority` + explicit width/height
   + `sizes`. CLS is 0. There's nothing further to optimise without
   moving the image into the static export, which v1 doesn't allow
   (post.cover_image_id is per-post and dynamic).

3. **Inline JSON-LD** is small (< 1KB each, two scripts). Not a
   blocker.

A11y ≥ 95 should now be hit thanks to this dispatch's modal focus-
restore + save indicator role fixes plus M3/M4 Tester carryover work.
SEO ≥ 100 should pass — meta description, canonical, robots, OG
images, JSON-LD all present.

## 8. Manual-verify path (5-step golden flow)

These are the human-driven checks that map to the M8-core binding
items. Assumes `pnpm next start` running on `:3000` with a published
post seeded.

1. **Login screen a11y.** Visit `/admin/login`. Open DevTools → Tab
   through the page from URL bar onward: focus order is Logo →
   Email → Password → Sign in (no skip, no trap). Run Axe (via the
   axe-devtools extension) — expect 0 violations.

2. **Dashboard delete-modal focus restore.** Visit `/admin`, sign
   in, ensure ≥1 post exists. Press Tab until you reach a row's
   Delete button. Press Enter / Space → modal opens, focus is on
   the slug-confirm input. Press Esc → modal closes AND focus is
   back on the same Delete button (NOT on `<body>`). Repeat with
   the Cancel button — same restore.

3. **Edit-form save announcement.** Visit `/admin/posts/[id]` for an
   existing post. Open VoiceOver (Cmd+F5 on macOS) or a screen-
   reader of choice. Type a character in the title field; wait 2s
   for autosave. Reader should announce "Saving…" then "Saved just
   now" (the role=status live region). Force a save error (e.g.,
   make the slug invalid for an unpublished post): reader announces
   the alert (assertive, interrupts).

4. **Image picker focus restore.** On the same edit page, click
   "Choose image" under Cover. Picker opens. Press Esc to close.
   Focus returns to the "Choose image" trigger (not `<body>`). Open
   it again, upload an image, fill alt, click Use. Picker closes;
   focus returns to the same Cover slot — now to the "Change"
   button (a different button, but in the same field).

5. **/og Cache-Control + /blog canonical via curl.**
   ```
   curl -I "http://localhost:3000/og?slug=<any-published-slug>"
   # expect: cache-control: public, max-age=0, s-maxage=86400, stale-while-revalidate=604800
   curl -s "http://localhost:3000/blog/?tag=defi" | grep -E '<link[^>]*rel="canonical"'
   # expect: <link rel="canonical" href="https://www.metaborong.com/blog"/>
   curl -s "http://localhost:3000/blog/?page=2" | grep -E '<link[^>]*rel="canonical"'
   # expect: same canonical to https://www.metaborong.com/blog
   curl -s "http://localhost:3000/blog/<slug>/" | grep -E '<link[^>]*rel="canonical"'
   # expect: canonical to https://www.metaborong.com/blog/<slug>/
   ```

## 9. Known gaps / open questions

1. **Render-blocking font @imports** in `app/globals.css`. Fontshare
   + Google Fonts loaded via `@import url(...)` at the top of the
   stylesheet. CSS `@import` is render-blocking AND serial — each
   font download waits for the previous to resolve before starting,
   so the LCP penalty compounds. **Fix:** convert to `next/font/local`
   for Satoshi (fontshare hosts the .woff2 set) and `next/font/google`
   for Space Grotesk + JetBrains Mono. Touches `app/layout.tsx` (font
   instantiation) and the homepage's heading typography, so it's
   out of the M8-core scope ("Do NOT touch app/page.tsx,
   components/sections/*, components/hero-orb/*"). Recommend a
   dedicated v1.5+ slice — `cms/v1.5-fe-fonts` or similar — that
   covers the homepage and admin/blog routes together. Likely
   Lighthouse Performance gain: +5 to +12 points on mobile.

2. **Live Lighthouse numbers not captured.** The session-wide hook
   (§1) made a long-running dev server unsafe — branch switches
   could move the page source mid-profile. Tester should run §7's
   procedure on a clean `cms/m8-core-fe-polish` checkout under
   `pnpm next start` (prod build, not dev) and record the four
   scores in their report.

3. **Live console-clean walkthrough not captured.** Same hook
   constraint. Static grep confirmed no client-side `console.*`
   calls in any FE-authored route or component; the only ones in
   the repo are server-side error logs (`app/api/admin/**`) and
   the homepage hero-orb's ErrorBoundary (out of scope). Tester
   should still walk every public + admin route in Chromium and
   report any unexpected warnings.

4. **`/blog/page.tsx` canonical** uses a relative path
   (`'/blog'`). Next 16's metadata system resolves this against
   `metadataBase` (already set in `app/layout.tsx` to
   `https://www.metaborong.com`), so the rendered tag will be
   `<link rel="canonical" href="https://www.metaborong.com/blog"/>`.
   Verified by reading Next docs; live curl deferred per #2.

5. **Status pill draft contrast is at the lower end.** Computed at
   ~5.3:1, comfortably above 4.5:1 but below 7:1 (AAA). If we ever
   tighten to AAA, draft pill text would need to deepen from
   `#676767` (gray) to a closer-to-`#404040` (dark) shade. Not
   blocking v1.

6. **Modal focus restore is "best-effort"** — if the trigger is
   removed from the DOM before the modal closes (e.g., the row was
   filtered out by an in-flight refresh), focus falls to the
   document body via the `document.body.contains(trigger)` gate.
   This is the right tradeoff for the dashboard delete flow (the
   row is intentionally gone after a successful delete). For other
   flows the trigger persists and focus restores correctly.

7. **`/blog` canonical doesn't disambiguate paginated listings as
   distinct documents.** A future move would be `rel="prev"` /
   `rel="next"` on `?page=N` URLs (Google deprecated them but Bing
   still uses them; some readers parse them). Out of v1 scope.
