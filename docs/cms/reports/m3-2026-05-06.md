## Defects (severity-ordered)

### HIGH

1. **`e2e/m2-posts-crud.spec.ts:91` regressed.** M3 replaced the content `<textarea>` in `components/admin/edit-post-form.tsx` with `<EditorShell />`. The M2 spec still does `await page.getByLabel(/content/i).fill(VALID_CONTENT)`, which times out 30s and aborts the rest of the M2 serial suite. Dispatch row B explicitly requires existing M2 e2e specs to still pass; the regression violates that gate.
   **Why HIGH:** binding row B is a stop-and-bounce condition; the FE handoff §6 claims "preserve other form fields, autosave, publish/unpublish, delete modal, beforeunload guard" but doesn't mention that the M2 e2e was also depending on the textarea.
   **Fix path** (~10 lines, FE owns):
     - Replace the M2 spec's textarea fill with a slash-menu insert sequence equivalent to what `e2e/m3-block-editor.spec.ts` already does, OR
     - Seed the post's `content_json` directly via `page.request.patch(...)` (the simplest equivalent — the M2 spec's intent is to verify the publish + public-render flow, not editor UX, which M3 covers).
   The FE handoff's §6 edge-case checklist row "Hydrate from existing M2-textarea posts via blocksToEditorState without data loss" was tested architecturally only; this defect is the load-bearing test for that hydration claim.

### MEDIUM

2. **A11y NIT: list-style toggle buttons have polluted accessible names.** In `components/admin/editor/inspector.tsx`, the inspector's `Field` component wraps its children in a `<label>`. For the List block, that means two `<button>` toggles ("Unordered" / "Ordered") sit inside the label. Playwright's accessibility tree (which mirrors what screen readers compute) resolves the Unordered button's accessible name as **"List style Ordered" [pressed]: Unordered** — the field's label text + the OTHER button's label text leak into each button's accessible name. A screen reader user toggling between list styles would hear something nonsensical.
   - Discovered while rounding out Row I — `getByRole('button', { name: 'Unordered', exact: true })` returns 0 even though the button is in the DOM. `getByText('Unordered', { exact: true })` correctly returns 1.
   - Functionally, the toggle works for sighted users (verified via text-locator click).
   - Fix: don't wrap non-form-control children in `<label>`. Either change `Field` to render a `<div role="group" aria-labelledby={…}>` for non-form-control content, or special-case ListFields to NOT use the wrapping `Field` component. ~3-line change.
   **Why MEDIUM, not HIGH:** the toggle is functionally green for ~95% of users; the regression is a11y-only. Not a v1 launch blocker, but a real assistive-tech defect that should be filed.

### LOW / NIT

3. **Inspector role-dropdown initial value shows the empty option for blocks with `role: null`.** The visible label is "— no role —" which is fine, but the underlying `<select>`'s value is `''`. Tester probe relied on `getByLabel(/^block role$/i).inputValue()` and got `''`. Not a defect — flagged so the FE knows the test selector for "role unset" is empty-string, not `null`.

4. **The inspector's role label uses `aria-label="Block role"` while the visible Field label is "Role"** (FE handoff §8 #2 acknowledged this). Playwright's `getByLabel(/^block role$/i)` matches the aria-label, while sighted users see "Role". Not a defect on the contract — `aria-label` overrides per ARIA — but worth aligning to one or the other for testability + readability.

5. **`/tmp/mb-test-pg` `.env.local` `\$`-escape requirement** still unfixed (M2 Tester report LOW 6). Same caveat applies — the bcrypt hash for `admin1234` was already in `.env.local` from M2 testing, so I didn't have to re-derive it.

### Out-of-scope (acknowledged, not blocking)

- **`e2e/m1-auth.spec.ts:53` page.goBack() flake.** Pre-existing, dispatch-flagged as out-of-scope. Same root cause noted in M1 + M2 reports. Likely fix: replace `goBack()` with `page.goto('/admin')`. **NOT downgrading the M3 verdict for this** per dispatch instructions.
