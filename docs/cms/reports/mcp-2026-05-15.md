## Buckets that surprised me

- **Positive — paranoia sweep for envelope leakage was already tight.**
  I expected to find at least one path where BE's `console.error(err)`
  branch left a serialised `MongoServerError` in the response body. It
  didn't. Every error path emits a `errorResponse(…)` / `jsonError(…)`
  with a hand-curated message; the raw error stays in `console.error`
  (server-side log only) and never reaches the wire. The `data.code`
  field surfaces the machine code; the `message` field is always
  human-readable. Code-walk corroborates: zero `JSON.stringify(err)` or
  `err.stack` references anywhere in `app/api/mcp/route.ts` or
  `lib/mcp/tools.ts`.

- **Positive — `content[0].type: 'json'` vs `'text'` wrapper.** The
  MCP standard documents a `text`-only wrapper. BE chose `json` (per
  handoff §4.8) so clients don't have to JSON.parse a string. I worried
  this would break older MCP clients, but the spec is permissive about
  the `type` field; clients that don't recognise `json` should fall
  back to `payload` (which BE also surfaces at the top level as a
  belt-and-suspenders). Smart call; document as a known compatibility
  consideration if a client ever surfaces a parser bug.

- **Positive — Playwright's APIRequestContext.post() JSON-encodes
  string `data` under content-type=application/json.** Caught me on
  the malformed-JSON e2e probe — my literal `'{not-json'` string was
  round-tripped through `JSON.stringify` by Playwright before being
  sent, so the wire payload was the VALID JSON string `'"{not-json"'`.
  The route's `req.json()` parsed it successfully, the envelope check
  fired second, and I got -32600 instead of the expected -32700.
  Workaround: pass a `Buffer.from(rawBody, 'utf8')` for malformed-body
  probes; Playwright passes Buffers through untouched. Documented
  in `e2e/mcp-regression.spec.ts:37-46` so the next Tester doesn't
  have to re-debug.

- **Operator-side surprise — admonition case-sensitivity.** Logged as
  N2. AI authors using `> [!Tip]` or `> [!TIP]` get a silent
  fall-through to a plain quote. The handoff didn't call this out, and
  the regex match in `lib/markdown/admonitions.ts` is exact-string. Easy
  fix; flagged as LOW because it's discoverable (Claude would see the
  rendered quote and learn to use lowercase) but it shouldn't be
  Claude's job to discover the casing convention.

## Coverage gaps I couldn't reach

- **Live `tools/call` round-trip against a real Mongo from the e2e
  spec.** My e2e suite's creds-gated tests skip cleanly without
  `E2E_MCP_TOKEN`, but they ALSO require a live Mongo to round-trip
  through `cms_create_draft` → `cms_get_post`. This Tester sandbox
  doesn't have a local mongod, so the 5 gated cases skipped. Vitest
  via `mongodb-memory-server` covers the same logic at the route
  level (`app/api/mcp/route.test.ts` + my `route.tester.test.ts`);
  the e2e is purely a smoke test for the production wire path. TPM
  should run the gated e2e in CI/staging where both Mongo and the
  bearer token are configured.

- **Real-world MCP client (Claude Code / claude.ai) end-to-end.** I
  did not configure Claude Code against `http://localhost:3000/api/mcp`
  and ask Claude to author a post. The JSON-RPC contract is
  airtight at the wire layer (Buckets A/C/G); whether `tools/list`
  rendering, `tools/call` argument shaping, and the `content[0].type:
  'json'` wrapper interact cleanly with Claude's MCP client is a
  TPM-owned manual smoke test. Recommendation: 5-minute manual probe
  before the production cutover.

- **Vercel timeout behavior under real Mongo cold-start load.**
  Logged in N3. Code-walk only; no live repro because the test
  environment didn't have a remote Mongo to simulate cold starts
  against.

- **Bearer-token rotation flow.** The handoff §6.2 documents
  per-environment tokens but I did not exercise an actual token
  rotation (mint new → swap in Vercel env → old token rejected, new
  token accepted). Procedural rather than functional; relies on the
  same `process.env.MCP_ADMIN_TOKEN` reload that all Vercel env
  changes inherit.

- **Image fetch from a real HTTP URL (`cms_upload_image` URL mode).**
  My Bucket D coverage exercises the base64 path heavily and the data:
  URL path once; the plain `http(s)://` path is covered only by the
  BE-authored `tools.test.ts` cases that don't actually hit the
  network (no test mocks `global.fetch`). The fetch + content-type
  check + size cap logic is exercised by the unit test but not by an
  HTTP probe. Adding a `nock` or `msw` integration to mock fetch would
  raise confidence; for v1 the existing coverage is adequate.
