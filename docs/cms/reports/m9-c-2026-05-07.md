## Defects (severity-ordered)

- **BLOCKER:** none.

- **HIGH:** none.

- **MEDIUM:** none.

- **LOW / NIT:**
  - **L1** — `pruneVariants` does not GC orphan `block_overrides[id]`
    entries when their underlying block has been deleted from
    `content_json`. Public-side render is unaffected (the resolver
    iterates current blocks only, never the override map). Storage
    cost is bounded by author behavior — small JSON strings per
    deleted-and-overridden block. **Recommended fix:**
    after computing `validation.parsedContent` in `runSave`, build
    `currentIds = new Set(parsedContent.map(b => b.id))` and pass it to
    `pruneVariants` so any `block_overrides[id]` whose id is not in
    `currentIds` is dropped. ~6-line change confined to
    `edit-post-form.tsx`. Suggested target: v1.6 (matches the v1.6 list
    already in handoff §9 items 4 / 5).
  - **L2** — Lighthouse not run in this environment; documented for
    TPM follow-up against a preview deploy. Not a code defect.
  - **N1 (NIT)** — handoff §9 item 1 already flags `geo_variant_regions`
    on `PostSummary` as an additive shared-schema change crossing
    BE-owned territory. Not held against M9-C since the dispatch
    explicitly took ownership and the field is purely additive; surface
    here only so the TPM tracks the BE-side reconciliation when the
    M9 super-merge lands.
