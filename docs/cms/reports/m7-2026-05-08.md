### Note on the dispatch's `page.route(verseodin.com)` directive

The dispatch §2 asks the spec to "Use Playwright `page.route()` to
intercept https://verseodin.com/api/mcp and reply with the JSON-RPC
fixture." That route is fetched **server-side** from
`lib/ai-readiness/client.ts:111` (Next.js dev server's Node runtime),
which Playwright's `page.route()` cannot intercept (it sees only
requests originating in the browser context). The pragmatic substitute,
documented in the spec's header docblock, is to mock at the FE→BE
boundary at `/api/admin/posts/[id]/ai-readiness`. The server-side fetch
to VerseOdin never fires when the boundary is fulfilled, so quota burn
is structurally impossible across the eight mocked cases. The
opt-in live-probe case (`E2E_VERSEODIN_LIVE=1`) is the only spec that
talks to VerseOdin; it issues exactly one `POST` against
`https://www.metaborong.com/`, matching the dispatch §2.10's "burns one
real quota unit" requirement.
