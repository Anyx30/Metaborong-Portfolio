## Verdict (one line)

**FAIL** — every binding backend row (C–I, W) is green and every binding
frontend row (J–V, except R) is green, **but** an a11y regression in the
new cover/og picker triggers (`<button id="f-cover">Change</button>` is
labelled by `<label htmlFor="f-cover">Cover image</label>`, polluting the
button's accessible name to `Cover image`) breaks the FE-shipped
`e2e/m4-images.spec.ts` at line 116 and would surface as
`button "Cover image"` to a screen reader. Same class of defect as M3
Tester report MEDIUM 2 (the M3 fix in `inspector.tsx` was not generalised
to `edit-post-form.tsx`'s new `ImageSlot`). One MEDIUM follow-up: the
in-editor live preview pane never renders cover or inline images
because `EditorShell → PreviewPane → PostView` is wired without a
`resolveImage` resolver.

### In-editor live preview pane gap (related to T but separate from the binding row)

T's binding compares standalone preview (`/admin/posts/[id]/preview`)
to public (`/blog/[slug]`) and they match. **But** the in-editor
preview pane (the right pane on `/admin/posts/[id]`) does NOT render
cover or inline images at all — it falls through to `<PostView />`'s
dashed-line placeholder for both. Root cause:

```
EditorShell  → PreviewPane(basePost, liveBlocks, liveOverlay)
PreviewPane  → <PostView post={previewPost} draftBanner />        // no resolveImage
PostView     → const cover = post.cover_image_id ? resolveImage?.(…) ?? null : null
             → cover === null AND post.cover_image_id !== null  →  <fallback dashed div>
```

`app/admin/posts/[id]/preview/page.tsx` and `app/blog/[slug]/page.tsx`
both wire `getImagesByIds(...)` server-side and pass the resolver.
`components/admin/editor/preview-pane.tsx:91` calls `<PostView />`
without one. FE handoff §7 step 4 explicitly claims "*The right pane
should now show the cover at the top of the post and the inline image
block via next/image*" — verifiably false in this dispatch. See
MEDIUM defect 2.

## Defects (severity-ordered)

### HIGH

1. **Cover / OG `<button id="f-cover">Change</button>` accessible name
   is `Cover image`, not `Change`.** Same root cause as M3 Tester
   report MEDIUM 2 (which was fixed inside `inspector.tsx` by replacing
   `<Field>` with `<div role="group" aria-labelledby={…}>`). M4's new
   `ImageSlot` in `components/admin/edit-post-form.tsx:638-672` reuses
   the parent `<Field id="f-cover">` (and `id="f-og"`) AND assigns the
   same id to the Change button (`<button id={id}>Change</button>` at
   line 654). HTML's labelable-element rule pairs `<label for="f-cover">`
   with the button, so screen readers and Playwright's role-based
   queries both compute the button's accessible name from the label
   text.

   - **Concrete consequences:**
     - `e2e/m4-images.spec.ts:116` fails:
       `getByRole('button', { name: /^change$/i }).first()` count = 0.
     - Screen readers will announce the cover Change button as
       "Cover image, button". (The Choose-image empty-state button
       currently overrides via `aria-label="Choose Cover image"` —
       ironically, the FE recognised the issue for the empty state
       and not for the populated state.)
   - **Probe evidence:**
     ```
     // walk: button#f-cover after upload
     attrs: { id: 'f-cover', textContent: 'Change', ariaLabel: null,
              ariaLabelledBy: null }
     label[for="f-cover"]: textContent = 'Cover image'
     getByRole('button', { name: /change/i }) count: 0
     getByRole('button', { name: /cover image/i }) count: 1
     ```
   - **Fix paths (FE owns; pick one, ~3 lines):**
     1. Drop `id={id}` from the Change button. The button is not a
        form control; the field's `<label htmlFor>` is decorative for
        the Change/Remove pair, since neither the trigger nor the
        thumbnail accepts a value to focus into. The label can lose
        its `htmlFor` when an image is set.
     2. Add `aria-label="Change cover image"` (and similarly for og)
        to the Change button so its accessible name is overridden
        regardless of the htmlFor association — same trick the empty-
        state button already uses.
     3. Apply the M3 fix to `<Field>` itself: when the field's child
        is not a single form control, render `<div role="group"
        aria-labelledby={…}>` instead of wrapping in `<label>`. This
        is the most general fix and removes the same trap from any
        future non-form-control consumer.
   - **Why HIGH:** breaks an FE-authored e2e spec the FE shipped this
     dispatch, AND is a real assistive-tech regression. Same severity
     I'd give a Cmd+S that cleared the form — works visually, but
     announces wrong over screen readers.

### MEDIUM

2. **In-editor live preview pane never renders cover or inline
   images.** `EditorShell → PreviewPane → PostView` is wired without a
   `resolveImage` resolver, so `<PostView />` always falls through to
   the dashed placeholder for both the cover and every inline `image`
   block. FE handoff §7 step 4 claims the right pane "*should now
   show the cover at the top of the post and the inline image block
   via next/image*" — verifiably false.

   - **Probe evidence:**
     ```
     editor preview <article> length: 1496  (vs public 4146)
     editor preview <img> count: 0
     ```
   - **Fix path (FE owns):** thread image rows through the prop
     boundary the same way the standalone-preview page does. One
     option: have `EditPostForm` keep a `resolvedImages: Map<id, Image>`
     in React state — seeded from the SSR-prefetched `initialCover`
     / `initialOg` plus any newly-uploaded image emitted by the
     picker — and pass `resolveImage={(id) => resolvedImages.get(id)
     ?? null}` through `<EditorShell />` → `<PreviewPane />` →
     `<PostView />`. Picker already updates the Map with each Use
     click; inline-block additions can re-use the empty-PATCH probe
     pattern that `ImageNodeView` already uses to fetch a single row.
   - **Why MEDIUM, not HIGH:** the binding row T compares standalone
     preview vs public, both of which are correctly wired. The
     editor-pane gap is *adjacent* — it's how the FE handoff
     describes the surface but isn't the dispatch's binding row. A
     reviewer can publish a post without ever needing the in-editor
     image preview, since the standalone preview is one click away.

### LOW / NIT

3. **`BLOB_READ_WRITE_TOKEN` provisioning still per-Tester.**
   `.env.local.example` documents the token + `\$`-escape caveat (BE
   handoff §3) but ships no working local-dev story. This Tester
   bridged with a `/tmp/blob-stub.mjs` HTTP server + the
   `VERCEL_BLOB_API_URL` SDK env hook (see "Files I touched"); future
   testers will hit the same blocker. Recommendation: add a one-liner
   to `BE handoff §3` linking to a "stand up a stub blob server"
   recipe, or ship a docker-compose with a stub blob container, OR
   provision a Vercel Blob dev token in shared secrets so future
   local-dev / Tester sessions just work.

4. **Progress bar not directly observable on tiny fixtures.** The
   picker wires `role="progressbar"` with `aria-valuenow` (line 571
   of `image-picker.tsx`), and `lib/api-client.postMultipart`'s XHR
   `progress` event is real (FE handoff §6). But the 68-byte 1×1 PNG
   fixture uploads in <50 ms, faster than my probe could poll the
   role count. Code is correct; the e2e fixture is just too small to
   observe the visual progress. Recommend adding a multi-MB binary
   fixture under `e2e/fixtures/` (FE owns) so future probes can
   actually see the progress bar tick. **Not a defect, just an
   observability gap.**

5. **M1 `e2e/m1-auth.spec.ts:78` `page.goBack()` flake.** Pre-existing,
   dispatch-flagged OOS, prior reports document. **NOT downgrading
   M4 verdict** per dispatch. Likely fix is the M2 Tester's suggestion:
   replace `page.goBack()` with `page.goto('/admin')` so the spec
   tests the gate's behaviour on navigation, not browser history.

6. **Standalone preview vs public: 32-byte article-HTML diff.** Public
   route's `<img>` carries `style="color:transparent;object-position:25% 75%"`;
   standalone preview's `<img>` carries `style="object-position: 25% 75%;"`.
   The `color:transparent` is `next/image`'s post-hydration FOIT
   guard. Structural DOM identical. **Not a defect** — same kind of
   normalised-away difference the M3 Tester report flagged in Step 3.

7. **M3 Tester MEDIUM 2 fix was scoped to `inspector.tsx`; the
   pattern (label-htmlFor pollution) isn't documented as a generally
   forbidden pattern.** Worth a one-line note in the FE-style guide
   (`agent-prompts.md` §3 EDGE CASES YOU OWN — A11y) so future agents
   don't recreate it.
