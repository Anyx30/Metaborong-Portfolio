### Setup notes that surfaced during Step 0
1. `.env.local`'s M1-era `ADMIN_PASSWORD_HASH` was the bcrypt of `hunter2` (BE handoff §6 default), but the dispatch's seeded e2e creds are `admin1234`. I regenerated the hash via `bcryptjs.hash('admin1234', 12)` and rewrote the `.env.local` line — local-only, gitignored. Flagging as MEDIUM defect 4 below: there's still no `pnpm db:seed:admin` (M1 Tester report MEDIUM 5), so each Tester dispatch has to re-derive this on its own.
2. **Important `.env.local` quirk:** Next.js loads `.env.local` through `@next/env` which performs `$VAR` expansion on values. Bcrypt hashes (which start `$2b$12$`) get **silently mangled** to just the trailing chars unless dollar signs are escaped as `\$`. The M1 handoff's recipe (`ADMIN_PASSWORD_HASH=$(node -e "...")`) writes the literal `$2b$12$…` which then loads as `.GHJUH5uX4YF…` and produces `401 invalid email or password`. Workaround: write `\$2b\$12\$…` in the file. I lost ~10 minutes here. Not a code defect, but worth a one-line note in BE handoff §6 / a `.env.local.example` to save future testers the same dead-end. Tracking as LOW defect 6.

## Defects (severity-ordered)

### MEDIUM

1. **`/admin/posts/[id]/preview` is NOT chrome-free / full-bleed.** The preview page renders inside `app/admin/layout.tsx`, which wraps everything in a sticky header (`Logo` + `SignOutButton`) and a max-width-1120 `<main>`. PRD §5.9.B explicitly calls for "full-bleed, chrome-free render of the draft — useful for previewing on a phone or showing a teammate at the founders' desk", and the dispatch's M-row repeats "full-bleed". On a small-screen / phone preview, the sticky admin header eats vertical space; on desktop the max-width visually restyles the post (different page width than the public render).
   **Fix:** add `app/admin/posts/[id]/preview/layout.tsx` that re-implements the `<html>` body slot without the admin chrome (or moves the auth gate into the preview page itself and removes the parent layout wrap). Pure FE change; ~10 lines; no API or schema impact.
   **Why MEDIUM, not HIGH:** the rendered post DOM (h1, blocks, FAQ, lede) is byte-identical to the public route — visual fidelity for the renderer ITSELF is preserved, which is the load-bearing claim from PRD §5.9 ("one renderer, three call sites"). The defect is the *frame around the post*, not the post.

2. **Driver-swap end-of-M2 cleanup not yet visible in this branch pair.** The hygiene PR (`e1ce5be` — `@vercel/postgres` → `pg`) is on `cms-dev` and present in both M2 branches, so this isn't a defect on the work I tested. Logging it for TPM awareness: M1 Tester report HIGH defect 1 should be marked resolved in the M1 report's audit trail when M2 merges.
